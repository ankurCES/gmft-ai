# ADR-0012 — v0.3.A implementation decisions

**Status:** Accepted (v0.3.0-A)
**Date:** 2026-06-17
**Deciders:** Ankur
**Supersedes:** none
**Related:** [ADR-0009](./0009-multi-agent-supervisor.md), [ADR-0010](./0010-v0.2-a.3-implementation-decisions.md)

## Context

v0.3.A is the run-polish slice that shipped the first stable
`run-polish` build (dist runtime fix, CI smoke test) and
completed the v0.2.A.2 / v0.2.A.3 supervisor work that the TUI
needed to actually fire in production. This ADR records the
A.1–A.4 implementation decisions that needed explicit calls.
None of them change the architectural promise in ADR-0009 / 0010
— the supervisor remains strictly silent and the chokepoint
remains the only hard gate.

## Decisions

### 1. `dist/` runtime is NodeNext + `.js` import suffixes, not a bundler

`apps/gmft` ships pre-built JS into `dist/` (one `cli.js` +
`bundle.js`) that CI and the `gmft` shim run directly with
`node`. The v0.2.A-era `dist/` was a hand-built `tsc` output
that used `import "./foo"` (no suffix), which Node's NodeNext
resolver rejects. The fix was to set `module: "NodeNext"`,
`moduleResolution: "NodeNext"`, and require `.js` suffixes on
every relative import (TypeScript resolves the `.ts` source at
compile time; the emitted `.js` references stay suffix-honest
for Node's runtime resolver).

**Why not a bundler (esbuild, swc, tsup)?** A bundler would
collapse `dist/` into a single `bundle.js` — which is what
phase 6 ended up shipping, but only after we cut `dist/cli.js`
first as a hand-built, debuggable target. The hand-built
NodeNext output is the minimal change that makes the existing
`pnpm build` artifact runnable. Adopting a bundler is a
follow-up that buys source maps and tree-shaking; it doesn't
unblock v0.3.A.

**Why not CommonJS?** CommonJS would have side-stepped the
suffix problem, but it would also have lost the ESM-only
imports from `@gmft/tools` (the tools package is pure ESM).
Cross-package ESM/CJS interop in Node 20+ is reliable only if
every link in the chain is ESM. Picking ESM at the leaf and
CJS at the app would have created a configuration tax we
didn't need to pay.

### 2. `SupervisorFireMarker` wiring is per-turn `eventIds` + session-wide `supervisorFires`

v0.2.A.2 shipped `SupervisorFireMarker` as a leaf component,
but the AgentApp-level wiring (deciding which transcript
message gets a marker) was deferred. v0.3.A.2 closes that
gap with two pieces of state:

- `currentTurnEventIdsRef: useRef<string[]>([])` — the agent
  loop's `for-await` pushes every `tool-call-request` /
  `tool-result` / `confirmation-needed` id into this ref as
  the events stream by. At the end of the turn, the ref is
  snapshotted onto the assistant `Message` as `eventIds`.
- `supervisorFires: useState<SupervisorFire[]>([])` — the
  session-wide accumulator. Every `supervisor-fire` event
  yields a `SupervisorFire` whose `targetEventId` points back
  to a runtime event we already saw; we also push that
  `targetEventId` into the per-turn collector so the
  assistant message for *this* turn carries the marker.

ChatTab's marker-rendering pass is then trivial: for each
`Message`, check if any `supervisorFire.targetEventId` is in
`Message.eventIds`, and if so, slot a `SupervisorFireMarker`
right after that message.

**Why not a single `Set<string>` of fired targets?** The
fire's payload (rule, kind, message) is per-fire, not per-id.
If the same `targetEventId` fires twice (e.g. loop-detected
re-fires after the operator resumes), we want to show two
markers with two different messages. The accumulator is an
array of `SupervisorFire`, not a set of ids.

**Why not a runtime lookup via a callback / ref?** That would
require ChatTab to be a child of AgentApp (it isn't — App
sits between them) or to take a callback down through three
layers of props. The state-array approach is the path of
least resistance that keeps the data flow unidirectional.

### 3. `--supervisor-model <id>` builds a second `createModel` call, not a model override in `withSupervisor`

`withSupervisor`'s v0.2.A.3 contract took a `model: LanguageModel`
opt and emitted a `SupervisorTurnRecord` with `modelUsed:
'agent-model'` hard-coded. v0.3.A.3 adds a `modelId?: string`
opt that flows into `SupervisorTurnRecord.modelUsed` (so the
postmortem record is no longer a lie when the caller passes a
different model). The AgentApp-level decision is: build a
*second* `createModel(...)` call for the supervisor that
reuses the primary's `provider` / `apiKey` / `endpoint` and
swaps in the override `model` id, and pass both `model` and
`modelId` to `withSupervisor`.

**Why not a per-model config object on `withSupervisor`?**
The supervisor wrapper is a pure helper. Forcing it to know
how to *construct* a model would couple it to the
model-factory's API and the API-key resolver. Passing a
pre-built `LanguageModel` plus a `modelId` string for the
record keeps the seam one-way: AgentApp knows the provider
ecosystem; `withSupervisor` only knows how to call
`generateText`.

**Why close the v0.2.A.3 "postmortem never fires" gap in
A.3 and not earlier?** The dist-runtime fix (A.1.1) had to
land first — until the CLI was actually runnable, no
production turn ever reached `withSupervisor`'s postmortem
branch, so the v0.2.A.3 "pass `model`" gap was latent. A.3
is the first time the postmortem fires end-to-end, which is
also the first time a `SupervisorTurnRecord.modelUsed` value
is ever visible to the user. Doing the `modelId` plumbing
in the same commit as the first live postmortem keeps the
record accurate from day one.

### 4. Audit log ships as a 4th tab, not a 5th modal or a transcript sub-pane

The plan called for a read-only `AuditLogTab` that paginates
the session's events, with default page size 50, filter by
`kind`, and color coding. The placement decision was between:

- **(a) A 4th tab** alongside Chat / Findings / Help. Chosen.
- (b) A sub-pane inside ChatTab. Rejected — ChatTab is
  already scroll-bound to the transcript; adding a second
  scroll surface would split input focus.
- (c) A modal overlay on Ctrl-A. Rejected — modal keybindings
  are reserved for destructive operations (Ctrl-C exit,
  slash-command pickers); a viewer is not destructive and
  shouldn't have a modal seam.
- (d) A `useAgent`-level event hook that the operator can
  wire to a `Console` dump. Rejected — defers the work to
  every consumer; the TUI is the consumer.

The tab approach matches the existing `TabBar` /
`TAB_ORDER` infrastructure (one new entry in
`TabId` / `TABS`, one new `activeTab === 'audit'` render
branch). The Tab / Shift-Tab cycle in `app-e2e.test.tsx`
was hard-coded to 3 tabs and had to be updated to 4 (forward
Chat→Findings→Help→Audit→Chat; reverse Help→Findings→Chat→
Audit) — a one-time cost, no infrastructure debt.

**Why not put the audit log *inside* FindingsTab?** Findings
are a chokepoint-side artifact (tool output, severity); the
audit log is a loop-side artifact (every yielded event). They
share no rendering code, no data shape, and no test fixtures.
Co-locating them would be cargo-cult UI.

**Why not paginate from the *agent-loop* side (push 50 at a
time)?** The agent loop is a generator; pagination is a
view-side concern. The loop yields every event in order, and
the accumulator (`useState<AgentEvent[]>`) appends. The
tab's local `page` state and `pageSize` are the only
pagination machinery. Splitting pagination across the loop
and the tab would couple the loop to the UI.

## Alternatives considered

**A. Bundler for `dist/`.** Rejected — buys source maps and
tree-shaking, doesn't unblock the A.1.1 fix. Deferred to a
follow-up phase (phase 6 ended up adopting a bundler for the
shipped `bundle.js`).

**B. CommonJS for `dist/cli.js`.** Rejected — cross-package
ESM/CJS interop is reliable only if every link is ESM.
CJS at the leaf + ESM at the app would have created a
config tax without simplifying the runtime.

**C. Runtime lookup via callback for the marker wiring.**
Rejected — would require threading a callback through three
layers of props. The state-array approach keeps data flow
unidirectional.

**D. Per-model config on `withSupervisor`.** Rejected —
couples a pure helper to the model-factory API. AgentApp
owns the provider ecosystem; `withSupervisor` only knows
how to call `generateText`.

**E. Audit log as a modal / sub-pane / useAgent hook.**
Rejected — modal keybindings are reserved for destructive
ops; sub-pane fights with ChatTab's scroll; hook defers the
work to every consumer.

## Consequences

- v0.3.0-A ships with 520 tests (508 pre-A.3 + 5 A.3 + 7 A.4).
- `dist/cli.js` runs end-to-end (`node ./dist/cli.js --help`
  smoke-tested in CI), closes the v0.2.A-era "no runnable
  artifact" gap.
- The supervisor postmortem fires in production for the
  first time; the `modelUsed` field on `SupervisorTurnRecord`
  is accurate from day one.
- The audit log is a first-class tab; operators can
  paginate, filter, and color-scan the loop's yield stream
  without leaving the TUI.
- The `Tab` / `Shift-Tab` cycle tests in `app-e2e.test.tsx`
  are now coupled to the current `TAB_ORDER` length. New
  tabs require a corresponding test update (caught and fixed
  in A.4).
