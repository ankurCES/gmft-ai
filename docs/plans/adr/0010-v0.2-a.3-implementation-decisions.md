# ADR-0010 — v0.2.A.3 implementation decisions

**Status:** Accepted (v0.2.0-A.3)
**Date:** 2026-06-12
**Deciders:** Ankur
**Supersedes:** none
**Related:** [ADR-0009](./0009-multi-agent-supervisor.md)

## Context

ADR-0009 set the high-level shape of v0.2.A (post-hoc supervisor
wrapper, 3 rules, end-of-turn postmortem, schemaVersion 2). This ADR
records the A.3 implementation decisions that needed explicit
calls. None of them change the architectural promise in ADR-0009 —
the supervisor remains strictly silent and the chokepoint remains
the only hard gate.

## Decisions

### 1. Postmortem is one LLM call, not streamed, with a 10s timeout

The 4-section prompt (WHAT WE TRIED / LEARNED / MISSING / NEXT STEP)
is a contract the TUI renders predictably. The postmortem is
generated via `generateText` (not `streamText`) and wrapped in
`Promise.race` with a 10s timeout. A failed or timed-out call
returns the empty string; the `SupervisorPostmortemCard` renders a
`(no postmortem — generator error)` placeholder so the card is
still meaningful. A turn with 0 fires skips the LLM call entirely
and uses a "quiet turn" fallback.

**Why not abort the turn on timeout?** The postmortem is
post-hoc — the agent's response is already done. Failing the
postmortem only fails the postmortem. A 10s ceiling protects
the session-log write from a stuck generator; the agent's actual
turn latency is unaffected.

**Why `generateText` and not `streamText`?** The card is a
single collapsed/expanded block. Streaming the postmortem would
add visual jitter (text appearing line-by-line) for no UX gain
— the user has already seen the agent's final answer.

### 2. `schemaVersion: 1 | 2` on `SessionRecord`, not per turn

The schema version is a session-level property. v0.1 logs default
to `1` (or absent, treated as `1`). v0.2 writes `2` on first
write. The migration is in `loadLog` — read `schemaVersion`,
parse the `supervisor` field only if `2`, redact secrets from
`supervisor.fires[].quote` and `supervisor.postmortem` on the way
in.

**Why not a per-turn version?** A per-turn version would imply
that old turns can be re-shaped on read, which is exactly what
schema migration is for. The session-level version captures the
intent: "v0.1 sessions don't have supervisor data; v0.2 sessions
do." Per-turn `schemaVersion` would be more granular but
unnecessary.

### 3. StatusRail Supervisor field has 3 explicit states

`quiet` (no fires, no postmortem), `fires` (1+ fires this turn),
`postmortem` (postmortem written). The state is updated by
`AgentApp` from the wrapped `lastFires()` / `lastPostmortem()`
calls — not from the supervisor directly. The pure helper
`renderSupervisorField` is the testable seam; the JSX wrapper
is exercised through the app-e2e tests.

**Why explicit states and not a count + a boolean?** The three
states are mutually exclusive and visually distinct (dim /
yellow / cyan). A count + boolean would compose into 4 states
(0/0, 0/1, 1/0, 1/1) but only 3 are valid. The 3-state union
is the minimal correct shape.

### 4. Transcript ⚠ markers shipped as `SupervisorFireMarker`

The plan called for a `Transcript.tsx` with `events[] +
firedTargets: Set<string>` props. That component doesn't exist
in the v0.1 app — events stream into `AgentApp` and the
supervisor-fire events are not currently rendered as a
transcript. The smallest renderable unit that satisfies the
contract is `SupervisorFireMarker` (one marker line per fire,
keyed on `targetEventId`). The AgentApp-level wiring (deciding
which messages get a marker) is separate follow-up work.

**Why not build Transcript.tsx?** A transcript component that
re-renders the event stream is a different design — it would
imply keeping the full event log in app state and rendering
it on every change. v0.1 doesn't do this. Building it now
would be scope creep and a behavior change the user didn't
ask for. The marker is the unit the user will see once the
wiring lands.

### 5. `SupervisorFire.kind` is the discriminant, not `rule`

The plan's example test for the marker used
`fire.rule: 'A' | 'B' | 'C'` with an `at: number` field. The
real type (defined in A.1's `supervisor-types.ts`) is a
discriminated union on `kind` (loop-detected / overclaim /
plan-issue) with no top-level `rule` / `at` fields. The shipped
test and component use the real shape; the marker maps
`kind` to a short `rule a/b/c` label for human consumption.

## Alternatives considered

**A. Stream the postmortem.** Rejected — visual jitter for no
UX gain; the agent's response is already in the message log.

**B. Per-turn `schemaVersion`.** Rejected — over-granular; the
session-level version captures the intent.

**C. Build Transcript.tsx as the plan described.** Rejected —
the v0.1 app doesn't render events as a transcript, and
building it now is scope creep. Shipped `SupervisorFireMarker`
as the smallest renderable unit; AgentApp-level wiring is
follow-up.

**D. Use `rule: 'A' | 'B' | 'C'` as the supervisor-fire
discriminant.** Rejected — the real type uses `kind` with
structured per-kind fields (e.g. `tool` + `count` for
loop-detected, `quote` + `evidence` for overclaim). A flat
`rule` field would lose the structure.

## Consequences

- v0.2.0-A.3 ships with 428 tests (374 baseline + 54 new).
- The postmortem is observable end-to-end: 1 LLM call per
  non-quiet turn, 10s timeout, never throws.
- The session log migrates v0.1 → v0.2 transparently on read.
- The TUI surface is split: `StatusRail` shows the state,
  `SupervisorFireMarker` shows individual fires, and
  `SupervisorPostmortemCard` shows the postmortem body.
  AgentApp-level wiring of the marker to the message stream
  is a follow-up.
