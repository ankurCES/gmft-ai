# ADR-0018 — v0.4-B AD attack tool gate

**Status:** Accepted (v0.4-B)
**Date:** 2026-06-19
**Deciders:** Ankur
**Supersedes:** none
**Related:** [v0.4 plan](../../2026-06-18-gmft-ai-v0.4.md) §4, [safety §10](../../safety.md)

## Context

The v0.4 plan §4 ships 5 AD attack tools: `impacket_gettgt`,
`impacket_kerberoast`, `impacket_psexec`, `impacket_secretsdump`,
`evilginx_phishlet_enable`. These are the most dangerous tools
gmft will ever ship — a single call to `secretsdump` against a
real domain controller produces NTLM hashes for every account
in AD, including the krbtgt. The same machine that can call
`secretsdump` is also a workstation the operator is logged into;
the failure mode is "we just dumped our own domain."

The wifi category has been doing this correctly since v0.3
(`evil_twin` is gated by `requiresElevation`, `destructive`,
and `typeToConfirm: 'attack'`). AD tools MUST be at least as
tight. But three AD-specific failure modes are not covered by
the existing wifi gate:

1. **`--scope` fan-out.** v0.3.B introduced `--scope` so
   `nmap` can run across N hosts in parallel. For recon that
   is fine. For AD attack tools, 50 hosts in parallel = 50
   simultaneous lateral-movement attempts = immediate
   detection + containment. The wifi gate does not block
   `--scope` (it has nothing to fan out to); AD tools need
   that block.
2. **The session host's own DC.** An operator running gmft
   on a domain-joined laptop must not be able to call
   `secretsdump` against `dc01.corp.example.com` without
   the machine flagging that "this target is your own DC."
   The wifi gate has no notion of "the host's own DC" — wifi
   is about the radio, not the realm.
3. **Credential exposure in transcript.** `secretsdump`
   outputs hashes in standard `user:rid:lmhash:nthash:::`
   form. These land in the session transcript JSONL, the
   audit log, and the PDF/JSON report. The v0.1.5g
   `redactSecrets` pass covers API keys, SSH keys, and
   env-var-shaped secrets — not NTLM hashes. The AD
   transcript must have a sibling redaction pass tuned
   for impacket output.

Without a written contract for these three, B.2 (the 5 tools)
will land with "the chokepoint does whatever the wifi gate
does" as the de-facto policy, which is a downgrade in two
of the three cases.

## Decision

### D.1 — Constraint A: same triple as `evil_twin`

AD tools use the existing destructive/elevation/type-to-confirm
gate unchanged. No new flag, no new env var beyond what the
wifi gate already reads:

| Tool flag | Runtime requirement |
| --- | --- |
| `requiresElevation: true` | `GMFT_ALLOW_ELEVATION=true` |
| `destructive: true` | y/n confirm in TUI |
| `typeToConfirm: 'ATTACK'` | operator types `ATTACK` literal |

The 5 AD tools will be declared in `packages/tools/src/catalog.ts`
with these three flags set. The chokepoint's existing
`checkDestructive` / `checkElevation` rules evaluate them; the
type-to-confirm UI already supports literal strings (added in
v0.3 for `evil_twin`).

### D.2 — Constraint B: `checkAdScope` rule blocks `--scope` for `category: 'ad'`

A new chokepoint rule `checkAdScope` is added at rule position
N (the rule-ordering decision is in §D.4 below). It is a single
predicate: **if `tool.category === 'ad'` and `args.scope` (or
`--scope` on the CLI) is set, reject the call.** The error is a
canonical-reason string so the operator sees the same message
in TUI, transcript, and audit log:

```
AD tools must be called one target at a time; --scope is not supported for this category.
```

The rule runs **before** `checkDestructive` and `checkElevation`
so the operator sees the category constraint first. This is a
deliberate reordering from the "confirm before constraint"
pattern of the wifi gate — for AD the category-level constraint
is the more informative error.

### D.3 — Constraint C: `checkDomainController` blocks the realm's PDC

A new chokepoint rule `checkDomainController` runs **after**
`checkAdScope` and **before** `checkDestructive`. It evaluates:

- If `GMFT_REALM_LOOKUP` is unset/false: skip the check (AD
  tools still pass through Constraint A; the operator is
  opting out of the realm-aware check).
- If `GMFT_REALM_LOOKUP=true`:
  1. Run `realm list --name-only` once per session and cache
     the PDC FQDN in the chokepoint session record.
  2. If the cache is empty (no realm, `realm` not installed,
     or krb5 not configured): reject ALL AD tool calls with
     the canonical reason `realm lookup enabled but no realm
     found; run 'realm list' to verify the host is domain-joined
     and Kerberos is configured`.
  3. If the cache has a PDC, compare `args.target` to it
     case-insensitively. A match rejects with `target matches
     the session's domain controller; this is blocked by default
     for AD tools`.

The cache is per-session (in the chokepoint session record) so
`realm list` is not re-run on every AD tool call. The first
`impacket_*` call in a session pays the lookup cost; subsequent
calls hit the cache.

`GMFT_REALM_LOOKUP` defaults to **false** to preserve the
"fail safe" property. When false, the operator gets no DC
protection, but the tools still work. When true, the operator
gets full protection, but on a host without `realm` configured
all AD tools break (loudly, with a remediation hint).

### D.4 — Rule ordering

The new chokepoint rule order for AD tool calls is:

1. `checkAdScope` (NEW) — `category: 'ad'` + `--scope` ⇒ reject
2. `checkDomainController` (NEW) — PDC match ⇒ reject
3. `checkElevation` (pre-existing) — `GMFT_ALLOW_ELEVATION`
4. `checkTypeToConfirm` (pre-existing) — `typeToConfirm` literal ⇒ type-then-confirm
5. `checkDestructive` (pre-existing) — `destructive` ⇒ confirm (y/n or literal)
6. `checkTarget` (pre-existing) — target format + RFC1918 + allowlist + session target
7. `checkRequiresSandbox` (pre-existing, v0.2.D) — host-without-sandbox block
8. `redactAdSecrets` (NEW, post-execution) — transcript redaction

The pre-v0.4-B aggregator fired `checkTarget` first. v0.4-B
moves `checkTarget` to step 6 so the canonical order
documented in `packages/core/src/chokepoint/rules.ts`
(`elevation → destructive → typeToConfirm → target → allow`)
is actually what the code runs. The contract is locked in
by `chokepoint.test.ts` → `describe('rule order (...)')`:

- `destructive beats target`: a `destructive+targetRequired`
  call with an illegal-char target still asks for confirmation
  (the operator may want to confirm the destructive call
  against a *known-bad* target to test detection; the target
  denial would skip the prompt).
- `elevation beats destructive`: `destructive+requiresElevation`
  denies without elevation, matching the pre-v0.4-B behavior.
- `elevation passes → target denies`: with `allowElevation: true`,
  `requiresElevation+targetRequired+bad target` denies on the
  target check, not on elevation.

The two new AD-category rules (`checkAdScope`,
`checkDomainController`) run before the four baseline rules
so the operator sees the category-level constraint first:

- `checkAdScope` runs **first** because the error is about a
  category-level constraint the operator may not have
  realized (e.g. "oh, `--scope` doesn't apply to AD tools").
  Surfacing that error first avoids burning a
  `checkDomainController` or `checkElevation` rejection on
  a call that would have been rejected by `checkAdScope`
  anyway.
- `checkDomainController` runs **second** because the DC
  check requires a `realm` lookup, which is more expensive
  than an env-var read. We do not want to pay that cost on
  a call that would have been rejected by `--scope` anyway,
  but we do want to pay it before the operator sees a
  `checkElevation` prompt (the realm check is more informative
  when the operator's own DC is the issue).
- `checkRequiresSandbox` runs **after** the destructive
  prompt so the user sees the confirm dialog before the
  v0.2.D host-without-sandbox block rejects the call.

This is the only place v0.4-B changes the chokepoint rule
order; it is documented in `safety.md` §10.4 and the
`v0.4 plan` §4. Per the versioning rule from `safety.md` §9,
a rule reorder requires an ADR — this is that ADR.

### D.5 — Constraint D: `redactAdSecrets` post-execution pass

A new function `redactAdSecrets` is added to
`packages/core/src/transcript/redact-ad.ts` and called from
the same write loop that already calls `redactSecrets`
(v0.1.5g). It matches:

| Pattern | Source | Replacement |
| --- | --- | --- |
| `\b[\w.-]+:[0-9]+:aad3b435b51404eeaad3b435b51404ee:[0-9a-f]{32}:::` | `secretsdump` SAM | `<redacted:ntlm-hash>` |
| `\\[\w.-]+:NTHASH:[0-9a-f]{32}` | `secretsdump` lsass/NTDS | `<redacted:lsass-nthash>` |
| `\$krb5tgs\$23\$*[\w.-]+\$[\w.-]+\$[\w./+=$-]+` | `kerberoast` TGS | `<redacted:kerberos-tgs>` |
| `[\w.-]+:[0-9]+:[0-9a-f]{32}:[0-9a-f]{32}:::` | generic SAM | `<redacted:ntlm-hash>` |

`redactAdSecrets` returns `{ redactedText, redactedFields: string[] }`.
`redactedFields` is logged into the audit event as a
`redacted_fields: string[]` field (additive — the v0.4.0-A.1
audit-log wire format is preserved for non-AD entries).

The `report_pdf` tool's `renderTranscript` step also calls
`redactAdSecrets` on the in-memory transcript before
rendering, so the PDF and JSON reports match the on-disk
transcript exactly. Without this, the redaction would be
bypassed by the report path (it reads from a
`TranscriptRow[]` array, not from the JSONL file).

## Implementation

### Catalog additions

`packages/tools/src/catalog.ts`:

```typescript
{
  name: 'impacket_secretsdump',
  category: 'ad',
  flags: ['destructive', 'targetRequired'],
  requiresElevation: true,
  destructive: true,
  typeToConfirm: 'ATTACK',
  binary: 'impacket-secretsdump',
  // ...
}
```

The same shape applies to the other 4 AD tools. The catalog
is the source of truth for `category: 'ad'` — the chokepoint
never has its own list of "AD tool names."

### Chokepoint rule additions

`packages/core/src/chokepoint/rules.ts`:

```typescript
export async function checkAdScope(
  call: ToolCall,
  session: ChokepointSession,
): Promise<CheckResult> {
  const tool = toolRegistry.get(call.toolName);
  if (tool?.category !== 'ad') return { allow: true };
  if (call.args.scope !== undefined || call.cliFlags?.includes('--scope')) {
    return {
      allow: false,
      canonicalReason: 'AD tools must be called one target at a time; --scope is not supported for this category.',
    };
  }
  return { allow: true };
}

export async function checkDomainController(
  call: ToolCall,
  session: ChokepointSession,
): Promise<CheckResult> {
  if (process.env.GMFT_REALM_LOOKUP !== 'true') return { allow: true };
  const tool = toolRegistry.get(call.toolName);
  if (tool?.category !== 'ad') return { allow: true };
  const pdc = await session.pdcCache.get();
  if (pdc === null) {
    return {
      allow: false,
      canonicalReason: "realm lookup enabled but no realm found; run 'realm list' to verify the host is domain-joined and Kerberos is configured",
    };
  }
  if (call.args.target?.toLowerCase() === pdc.toLowerCase()) {
    return {
      allow: false,
      canonicalReason: 'target matches the session\'s domain controller; this is blocked by default for AD tools.',
    };
  }
  return { allow: true };
}
```

### Reordering

`packages/core/src/chokepoint/chain.ts`:

The rule chain moves from
`[checkTarget, checkElevation, checkDestructive, confirm]`
to
`[checkTarget, checkAdScope, checkDomainController, checkElevation, checkDestructive, confirm]`.

The reorder is a single line in the chain definition. Tests
in `packages/core/test/chokepoint-chain.test.ts` are
updated to reflect the new order for AD tools; the existing
non-AD test cases keep their behavior.

### Redaction

`packages/core/src/transcript/redact-ad.ts` (new file,
~60 lines) implements `redactAdSecrets`. It is called from
`packages/core/src/transcript/write.ts` after the existing
`redactSecrets` call. The `report_pdf` tool's
`renderTranscript` imports it from the same module.

### Tests

- `packages/core/test/chokepoint-rules-check-ad-scope.test.ts`
  (NEW, ~6 tests): each AD tool + `--scope` ⇒ reject; each
  AD tool without `--scope` ⇒ allow; non-AD tool with
  `--scope` ⇒ allow; `category: 'ad'` is the gate, not the
  tool name.
- `packages/core/test/chokepoint-rules-check-dc.test.ts`
  (NEW, ~6 tests): `GMFT_REALM_LOOKUP=true` + realm returns
  PDC + AD tool against PDC ⇒ reject; AD tool against
  non-PDC ⇒ allow; `GMFT_REALM_LOOKUP=false` ⇒ skip; realm
  not found + `GMFT_REALM_LOOKUP=true` ⇒ reject all AD tools.
- `packages/core/test/transcript-redact-ad.test.ts`
  (NEW, ~5 tests): each of the 4 regexes matches the
  expected impacket output; the audit-log entry gets
  `redacted_fields: ['ntlm-hash', ...]`.
- `packages/core/test/chokepoint-chain.test.ts`
  (UPDATE): the order-of-rules test for AD tool calls now
  asserts `checkAdScope` runs before `checkElevation`.
- `packages/tools/test/catalog-ad.test.ts` (NEW, ~3 tests):
  all 5 AD tools have `category: 'ad'`, `requiresElevation:
  true`, `destructive: true`, `typeToConfirm: 'ATTACK'`.

Test budget impact: ~20 new tests. Net workspace target
post-v0.4-B.7: **≥ 900** tests.

## Alternatives considered

### Alt 1 — Reject AD tools entirely

Ship only the wifi + web + binary tools; defer AD to a
hypothetical v0.5.

**Why not:** the v0.4 plan §4 commits to AD tooling in
v0.4-B. Deferring pushes the threat-model work to a later
release, where the same constraints will have to be
re-derived. The threat model needs to land now so the
chokepoint's evolution has a written contract.

### Alt 2 — Use the same `typeToConfirm: 'attack'` as `evil_twin` (lowercase)

The wifi gate uses `'attack'` (lowercase) for `evil_twin`.
We could reuse that literal for AD tools to avoid a new
string.

**Why not:** the two categories are different, and a
distinct literal (`'ATTACK'`, uppercase) gives the
operator a visual cue that they are about to confirm a
different category of tool. The TUI confirm box already
uppercases user input for comparison, so this is a
one-character change in the catalog.

### Alt 3 — Run `realm list` on every AD call (no cache)

Simpler — no per-session cache, no staleness concerns.

**Why not:** `realm list` shells out to `realm`, which
in turn reads `/etc/krb5.conf`, possibly talks to AD
LDAP, and prints results. It is not fast enough to run
on every tool call. Caching per-session is the right
granularity: a session is at most a few hours, and
re-running `realm list` once per session is the
acceptable cost.

### Alt 4 — Block AD tools on hosts that have a real `/etc/krb5.conf`

If the operator is on a domain-joined host, refuse to
even load the AD tools.

**Why not:** this is the inverse of the design. The
operator who has *deliberately* joined a domain is
exactly the one who should be able to use the AD tools
*against a target other than their own DC*. Blocking at
the realm level removes the most common legitimate use
case. Blocking at the *target* level (Constraint C)
keeps the tools usable.

### Alt 5 — Put redaction in the chokepoint rather than the transcript

A chokepoint rule that strips hashes from the in-memory
`ToolCallResult` before the transcript is written.

**Why not:** the chokepoint is a *gate*; mutating tool
output from a gate is a layering violation. The v0.1.5g
`redactSecrets` pass already lives in the transcript
write loop, and the AD redaction is a sibling pass in
the same loop. Keeping redaction in the write loop also
means the `report_pdf` path can reuse the same function
without re-deriving the regexes.

## Consequences

### Positive

- AD tool calls are auditable in the same log format as
  every other gmft tool. The audit log gets a new
  optional field (`redacted_fields`) that operators can
  grep for.
- The DC block + the `--scope` block are both expressed
  in canonical-reason strings, so the operator sees a
  consistent error regardless of whether the call came
  from the TUI, the CLI, or a programmatic client.
- `realm list` runs at most once per session.
- Reordering the rule chain is documented here and in
  `safety.md` §10.4. Future rule additions know to add
  *before* `checkAdScope` if they should short-circuit
  the AD-specific checks.

### Negative

- The rule chain is now longer (4 → 6 steps for AD tool
  calls). The chain code is shared with non-AD tools, so
  every non-AD call also pays the cost of evaluating
  `checkAdScope` and `checkDomainController` (both of
  which short-circuit on `category !== 'ad'`). The cost
  is one extra hash lookup and one extra env-var read
  per call. Acceptable.
- `GMFT_REALM_LOOKUP=true` on a host without `realm`
  breaks AD tools. The error message is loud and gives
  a remediation hint, but it is still a regression for
  any operator who set the env var assuming it would
  "just work."
- The `redactAdSecrets` regexes are a maintenance
  burden: impacket output formats can change between
  versions, and a regex that misses a new format silently
  leaks hashes. The test suite covers the 4 known
  formats; new formats need a new test and a new regex.
  This is acknowledged in the §10.3 hardening checklist
  ("a report with zero redactions and successful AD tool
  invocations is a bug").

## Open follow-ups

- **v0.4-B.6 (catalog doc):** `docs/tool-catalog.md` must
  list the 5 AD tools with the elevation, destructive, and
  type-to-confirm flags. This ADR's catalog example is
  the seed.
- **v0.5+ (LLM-judge for AD tool plans):** the v0.4-A.2
  `judgePlanQuality` work could be extended to refuse
  AD-tool plans that do not have a corresponding `--target`
  argument resolved (i.e. where the LLM is about to spray
  credential a subnet). This is a future feature and is
  not in v0.4-B scope.
- **v0.5+ (offline PDC discovery):** `realm list` requires
  the host to be domain-joined. For operators on non-joined
  hosts (e.g. a MacBook running gmft against a remote DC),
  `GMFT_REALM_LOOKUP` is effectively a no-op. A future
  ADR could add an opt-in `GMFT_PDC_FILE=/path/to/pdc.txt`
  that reads the PDC FQDN from a file the operator
  maintains. Not in v0.4-B scope.
- **v0.5+ (multi-DC forests):** the current check only
  blocks the realm's PDC. A multi-DC forest has multiple
  DCs, and an operator might want to block *all* of them
  (resolved via `dig SRV _ldap._tcp.dc._msdcs.<realm>`).
  Not in v0.4-B scope.
