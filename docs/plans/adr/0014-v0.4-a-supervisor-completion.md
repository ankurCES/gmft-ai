## Context

ADR-0009 introduced a strictly-silent post-hoc supervisor and deferred
two triggers ("3: overreach" and "5: risk escalation") to a later
release. v0.4-A is that release. The v0.4 plan Â§A.1 scope was to add
both triggers. During the doubt-driven-development pre-implementation
review (2026-06-18), the reviewer surfaced that:

1. The proposed Trigger 3 (overreach) assumed `args.target` was the
   target-bearing field on destructive tool calls. Reading the actual
   tool registry, this assumption is wrong: the destructive surface
   uses domain-specific fields (`url`, `targetBssid`, no-field at
   all on `shell_exec`, per-step `args` on `attack_chain`). The
   supervisor cannot reliably read the target out of `args`.
2. The proposed Trigger 5 (risk escalation) was poorly distinguished
   from Rule C.1. The plan wording said "went from read-only to
   destructive within a turn without any intervening recon" â€” but
   that is literally what Rule C.1 already does. The implementation
   attempt was muddled.
3. `AgentApp.tsx:713` passes `chokepointSessionTarget: undefined` to
   `withSupervisor` (with a comment that this is safe because Rule
   C.3 has no `targetRequired` tools to consult). Trigger 3, as
   proposed, would have been production-dead for this reason as
   well.

## Decision

### Trigger 3 (overreach) is the chokepoint's job â€” NOT added to the supervisor

The chokepoint already enforces "no destructive call against an
out-of-scope target" via `checkTarget` in
`packages/core/src/chokepoint/rules.ts`. The supervisor does not
duplicate this check. Duplicating it would:

- Add a parallel rule that reads `args` shape that doesn't exist on
  the actual destructive surface (sqlmap uses `args.url`, evil-twin
  uses `args.targetBssid`, shell_exec has no target field at all).
- Drift from the chokepoint's authoritative decision the moment one
  of them is updated.
- Add false positives (supervisor fires "out of scope" while the
  chokepoint has already denied the call â€” operator sees two âš 
  markers for one denial).

ADR-0009's principle stands: "The supervisor cannot block tool
calls, override the chokepoint, or prompt the user. The chokepoint
stays the only hard gate." Trigger 3 belongs in the chokepoint.

**The v0.4-A plan's Â§A.1 wording on Trigger 3 is therefore
inoperative.** Trigger 3 is **not** shipped in v0.4-A. The v0.4
plan spec at `docs/plans/2026-06-18-gmft-ai-v0.4.md` Â§A.1 is
updated to reflect this (see Â§3 follow-ups below).

### Trigger 5 (risk escalation) reframed as "destructive as first tool of turn"

The supervisor **does** add a new rule â€” but reframed as a *stricter*
gate than Rule C.1:

- **Rule C.1 (existing):** "destructive tool called with 0 prior
  recon calls in the turn AND at least 1 prior tool of any kind."
  Suppressed when the first tool of the turn is destructive (see
  `supervisor-rules.ts:339` â€” the `toolsCalledThisTurn > 0` gate).
- **Rule E (new):** "destructive tool called as the **first** tool of
  the turn." Catches the case C.1 deliberately skips.

The two rules are not duplicates â€” they cover different gaps:

| Gate | Rule C.1 | Rule E |
|---|---|---|
| Destructive + no recon in turn | âś“ | âś“ (subset â€” recon=0 implied by first tool) |
| First tool of turn is destructive | âś— (deliberately) | âś“ |
| 2nd+ tool of turn is destructive (with prior recon) | âś— | âś— |

Rule E's fire is a new `risk-escalation` variant in the
`SupervisorFire` discriminated union. It is wired into the wrapper
**BEFORE** `observeRuleC` so the gate reads the pre-call counter
(`state.ruleC.toolsCalledThisTurn === 0`). Wiring it after C would
cause the gate to read `=== 1` (post-increment) and over-fire on
2nd-and-later destructive calls in the same turn.

### Why this matters

"First tool of turn is destructive" is a distinct safety signal from
"destructive without recon in turn." A user opening a session with
`gmft --target example.com` and the agent immediately firing `nmap
-p- example.com` is fine; firing `sqlmap http://example.com/` as the
literal first action is "escalation without hesitation" â€” a
different kind of plan-quality issue that C.1 was designed *not* to
flag. Rule E fills that gap with one fire shape, one wire-format
extension, and one additional wrapper wiring.

### Rule code â€” `packages/core/src/agent/supervisor-rules.ts`

New function `observeRuleE`, mirroring the `observeRule<X>` pattern:

```typescript
// Rule E â€” Risk escalation (destructive as first tool of turn)
//
// Fires when a destructive tool is the FIRST tool of the turn.
// This is a stricter gate than Rule C.1, which deliberately skips
// the first tool. The two rules cover different gaps:
//   C.1: "destructive without recon in turn" (any tool came first)
//   E:   "destructive as the very first action of the turn"
// Wired into the wrapper BEFORE observeRuleC so the gate reads the
// pre-call counter. (After C runs, the counter is post-incremented to
// 1 and the gate would over-fire.)
export function observeRuleE(
  state: SupervisorState,
  event: AgentEvent,
): { state: SupervisorState; fire?: RiskEscalationFire } {
  if (event.type !== 'tool-call-request') return { state };
  const { name, id, flags } = event;
  const isDestructive = flags?.includes('destructive') ?? false;
  if (!isDestructive) return { state };
  if (state.ruleC.toolsCalledThisTurn !== 0) return { state };
  const fire: RiskEscalationFire = {
    kind: 'risk-escalation',
    tool: name,
    firstToolOfTurn: true,
    advice: `Supervisor: you opened this turn with a destructive tool (\`${name}\`) â€” no prior recon, no read-only inspection, no plan statement. Recon first, then act.`,
    targetEventId: id,
  };
  return { state, fire };
}
```

## Alternatives considered

**A. Ship Trigger 3 (overreach) anyway, by adding a per-tool target
extractor to the supervisor.** Rejected: every new tool with a
destructive flag would need a per-tool extractor maintained in the
supervisor package â€” a clear violation of the registry-as-source-of-
truth principle. The chokepoint already has this extractor in
`checkTarget`; the supervisor would be reading a parallel copy.

**B. Ship Trigger 5 (risk escalation) as a refinement of Rule C.1
rather than a new rule.** Rejected: changing C.1's gate from
`toolsCalledThisTurn > 0` to `>= 0` would cause C.1 to fire on every
single-turn destructive session â€” a high false-positive rate, and
it changes the v0.2.A-tested behavior of C.1. A new rule with its
own advice text is the right blast radius.

**C. Wire Rule E AFTER observeRuleC and read `toolsCalledThisTurn === 1`.**
Rejected: this couples Rule E's correctness to C's wire order. A
future maintainer who re-orders the wrapper (e.g. to put C first for
"first-match-wins" performance) silently breaks E. Reading the
pre-call counter and wiring E first is the safer ordering.

## Consequences

- `SupervisorFireRecordSchema` grows by 1 Zod variant (additive).
  v0.3-C audit log readers (which already accept
  `SupervisorFireRecordSchema`) continue to parse v0.4-A logs.
  Old log files (v0.2.A / v0.3.A / v0.3-C) parse unchanged â€” they
  just don't contain `risk-escalation` fires.
- `supervisor.ts` grows by 1 wiring line + 1 list element. No
  semantic change to existing rules.
- The supervisor gains a new fire shape. Existing TUI rendering
  (`apps/gmft/src/components/StatusRail.tsx`) renders fires by
  `fire.kind` and falls back to a generic âš  marker; the new
  `risk-escalation` kind renders as a generic âš  unless the TUI
  is updated to add a kind-specific icon. A kind-specific icon
  is a v0.4-A.4 (CLI surface) follow-up.
- The `risk-escalation` fire is suppressed for v0.3-A replay because
  the schema-version field on the session log would not include
  it. Operators upgrading from v0.3.A keep all existing fires
  intact.
- The "first tool of turn" semantics are sensitive to the order in
  which the wrapper invokes the rule. ADR-0014 fixes the order
  (A â†’ E â†’ C â†’ B). The docstring on `observeRuleE` calls this out
  in the file header.
