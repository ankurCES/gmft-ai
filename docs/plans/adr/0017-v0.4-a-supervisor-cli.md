# ADR-0017 — /supervisor slash command + tab-completion

- Status: Accepted
- Date: 2026-06-18
- Plan: v0.4-A §A.4 (Supervisor CLI surface)
- Builds on: ADR-0014 (supervisor completion), ADR-0015 (LLM judge for plan quality), ADR-0016 (supervisor fires in audit log)

## Context

The supervisor (ADR-0014) wraps `runTurn` and exposes two accessors
on the `withSupervisor` return value: `lastFires()` (the array of
`SupervisorFire` events for the most recent turn) and `lastPostmortem()`
(the LLM-generated postmortem prose, or `undefined` if generation
was skipped or failed).

Those accessors were only reachable from inside the `runTurn` call
site, which lives in `AgentApp`'s submit closure. The operator had no
in-app way to inspect them after a turn completed — the only path
was reading the on-disk session log directly, which is verbose and
not the operator's primary workflow.

The slash-command surface (`/audit`, `/report`, `/help`, etc.) was
the natural place to add this. The plan for v0.4-A §A.4 called for a
`/supervisor` command that surfaces the supervisor's state for the
most recent turn.

## Decision

Add a new `handleSupervisor` branch to `apps/gmft/src/session/commands.ts`
that reads the supervisor's last-fires + last-postmortem via a ref the
component stores after each turn, and renders a chat reply via a pure
formatter function.

### Why a ref (not a callback closure)

`AgentApp`'s submit handler is a closure that has direct access to
the wrapper, but the slash-command dispatcher lives outside that
closure. Three options:

1. **Refactor `runTurn` to be slash-command-aware.** Rejected: this
   couples the agent loop to the chat surface.
2. **Recompute the snapshot on every `/supervisor` call** by replaying
   the last turn. Rejected: re-runs the supervisor (and the LLM judge
   in A.2), which is expensive and changes the wrapper's state.
3. **Store the wrapper in a `useRef` after each turn**, and expose a
   `getSupervisorSnapshot` callback that reads
   `lastFires()` / `lastPostmortem()` from the ref. **Chosen.**

The ref approach is cheap (one field read + two method calls), gives
the slash-command dispatcher a stable interface, and doesn't change
the wrapper's contract.

### Why a pure formatter

`formatSupervisorSnapshot` is a pure function `(SupervisorSnapshot) => string`
with no I/O, no React, no side effects. This is the same pattern used
elsewhere in the slash-command surface (e.g. `formatAuditHead`,
`formatFindings`): the dispatcher parses the slash arguments, calls
the formatter, and wraps the result in a `Msg`. Three benefits:

1. **Testable in isolation** — the 7 new unit tests in
   `slash-commands.test.ts` exercise the formatter directly without
   spinning up the React tree.
2. **Reusable for the CLI** — if a future `gmft supervisor` subcommand
   is added, it can import the same formatter and avoid duplicating
   the layout logic.
3. **Snapshot-stable** — the exact rendered string is asserted in
   tests, so any layout regression breaks the build.

### Three subcommands, one default

The dispatcher accepts three subcommands: `fires`, `postmortem`, and
the default (both). Each maps to a different view-shaped snapshot
that the formatter renders:

| Subcommand        | `view.fires`       | `view.postmortem`         | Renders                        |
| ----------------- | ------------------ | ------------------------- | ------------------------------ |
| `/supervisor`     | snapshot.fires     | snapshot.postmortem       | "Last turn: N fire(s)" + Fires list + Postmortem section |
| `/supervisor fires` | snapshot.fires   | omitted                   | "Last turn: N fire(s)" + Fires list only |
| `/supervisor postmortem` | `[]`         | snapshot.postmortem       | Postmortem section only (no fires header) |

The default is "show me everything from the last turn" — the
operator's natural post-mortem workflow. The two subcommands are
for operators who already know which half they want (e.g. "did
anything fire?" or "what was the LLM's reasoning?").

The `postmortem`-only subcommand passes `fires: []` to the
formatter. The formatter's layout rules skip the fires header
when `fires.length === 0` AND `postmortem !== undefined`, so
only the postmortem section renders. This avoids the dispatcher
needing a "suppress fires header" flag.

### Error visibility

The postmortem schema has two fields: `postmortem` (prose body)
and `postmortemError` (string when generation failed, e.g. LLM
timeout, model 503). The formatter's body-rendering branch checks
`postmortem` first, then falls back to
`(postmortem generation failed: <postmortemError>)` if `postmortem`
is empty but `postmortemError` is set. This way operators see
*why* the postmortem didn't generate, instead of an empty section.

### Tab-completion

`apps/gmft/src/session/tab-completion.ts`'s `SLASH_COMMANDS` array
already had `/audit` and `/report` listed; `/supervisor` was the
missing one. A.4 added it. As a drive-by fix, the pre-existing
omission of `/audit` from the same array (which the A.4 plan
caught) is also fixed.

### Severity narrowing in `SupervisorFireMarker`

A.1 added a new `RiskEscalationFire` variant with
`firstToolOfTurn: true`. A.4's first attempt to compile the
`apps/gmft/src/ui/components/SupervisorFireMarker.tsx` change
uncovered a latent bug: the marker assumed `f.severity` was
always present, but the new variant doesn't carry a `severity`
field. Fixed by narrowing on `'severity' in f` for the marker
UI, matching the same narrowing pattern in the formatter.

## Consequences

- Operators can inspect supervisor state from the chat pane via
  `/supervisor`, without leaving the app or parsing the session
  log manually.
- The formatter is unit-tested (7 tests) and reusable for any
  future CLI surface.
- `AgentApp` has one new ref (`wrappedSupervisorRef`) and one new
  callback (`getSupervisorSnapshot`) — minimal surface change.
- Tab-completion is now consistent across all three top-level
  commands (`/supervisor`, `/audit`, `/report`).
- The `SupervisorFireMarker` UI now handles all four `SupervisorFire`
  variants correctly (A.1's `risk-escalation` variant was previously
  rendering with an undefined severity; fixed in A.4).

## Alternatives considered

1. **Read supervisor state from the session log file on disk.**
   Rejected: chat is the operator's primary surface, the session
   log is verbose JSONL, and re-parsing on every `/supervisor`
   call is wasteful.
2. **Subscribe to supervisor fires via an event channel and store
   them in a Zustand-style store.** Rejected: the supervisor
   already exposes `lastFires()`; adding a parallel store would
   duplicate the state and risk drift.
3. **Render supervisor state in a dedicated right-rail panel**
   (like the existing `StatusRail`). Rejected: the rail already
   shows per-turn status; supervisor state is "post-mortem
   review" content, which is a slash-command-shaped workflow
   (operator-initiated, on-demand), not a passive status
   display.
