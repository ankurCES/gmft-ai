# ADR-0013 — v0.3.C audit log design

**Status:** Accepted (v0.3.0-C)
**Date:** 2026-06-17
**Deciders:** Ankur
**Supersedes:** none
**Related:** [ADR-0006](./0006-chokepoint-first-tool-dispatch.md), [ADR-0009](./0009-multi-agent-supervisor.md)

## Context

v0.3.A introduced a `supervisor` that fires when the agent loop
detects runaway behavior — but the only post-hoc record of *what
the chokepoint actually decided* was the in-memory event stream
the TUI saw, plus the findings JSON the supervisor wrote when a
fire resolved. There was no tamper-evident log of the chokepoint's
decisions, the runner's mode, or the session lifecycle. v0.3.C
adds that log: a hash-chained, HMAC-signed JSONL at
`$XDG_CONFIG_HOME/gmft/audit/audit.jsonl` with a CLI to verify,
browse, and follow it.

The hard requirement: any post-mortem that asks "did the
chokepoint let tool X through at time T?" must be answerable
from a file on disk — not from a TUI screenshot, not from an
in-memory buffer, not from the supervisor's postmortem card.
The TUI's `AuditLogTab` reads from the *agent event stream*,
which is observability; the audit log is *evidence*.

## Decisions

### 1. Append-only, hash-chained JSONL

Each line is a JSON-encoded `AuditEvent` with the shape:

```ts
interface AuditEvent {
  ts: string;                     // ISO 8601, set by the writer
  kind: AuditEventKind;           // see §3
  prevHash: string;               // 64 hex chars; GENESIS (all zeros) for line 1
  hash: string;                   // 64 hex chars; HMAC-SHA-256 of canonicalForm(...)
  payload: Record<string, unknown>; // NOT included in the hash
}
```

The `hash` covers `ts + kind + prevHash + payload` (via
`canonicalForm` in `audit/types.ts`, which sorts object keys
recursively so verification reproduces the exact same bytes
regardless of insertion order). The chain: line N's `prevHash`
equals line N-1's `hash`; line 1's `prevHash` equals
`GENESIS_PREV_HASH` (64 zero hex chars).

**Why HMAC, not plain SHA-256?** The chain proves *order* and
*absence of tampering* with SHA-256 alone, but a hash chain
written by an attacker with write access to the file can be
re-written end-to-end. An HMAC keyed by a file the attacker
can't read makes re-writing infeasible without the key.

**Why one key, not per-event keys?** A single root key is
simpler to back up (`gmft audit backup-key` is a single command,
not a key tree) and the threat model is "post-hoc verification of
a known-good key" — not "rotating compromised event keys". Key
rotation is a v0.4 follow-up.

### 2. Key storage: 0600 file in the audit dir, with a secret-store backup

The HMAC key is a 32-byte random buffer, stored as
`$XDG_CONFIG_HOME/gmft/audit/audit.key` with mode 0600. The
`getOrCreateHmacKey` function:

- On first run, generates 32 random bytes and writes them with
  mode 0600.
- On subsequent runs, reads the file (and stat-checks mode —
  corrects to 0600 on first append after a chmod tampering).
- Optionally mirrors the key to the user's secret store (the
  same store the LLM API keys live in) under the name
  `audit.hmac_key` — the `audit backup-key` and `audit
  restore-key` CLI commands move between file and store.

**Why not just the secret store, no file?** The file is the
authoritative source because it has filesystem permissions (the
secret store typically doesn't, on Linux). The secret store is
the *backup* — if the file is lost, the log is no longer
verifiable (chain intact, but no key to recompute hashes
against), so a copy belongs in the secret store from the start.

**Why 0600 specifically?** The key is the only thing standing
between a stolen log file and a complete re-write. The
chmod-on-first-append is a belt-and-suspenders check (a
previous version of the file might have been 0644 after a
backup restore).

### 3. Event kinds and the writer's contract

The `AuditEventKind` union is intentionally narrow — it's the
*security-relevant* set, not "every agent event":

```ts
type AuditEventKind =
  | 'session-start'
  | 'session-end'
  | 'tool-call'
  | 'tool-result'
  | 'chokepoint-decision'
  | 'runner-mode'
  | 'onboard';
```

`AuditWriter.append(kind, payload)` is the only way to add a
line. It:

1. Serializes the event with `ts = new Date().toISOString()`
2. Computes `prevHash` from the last line on disk (or
   `GENESIS_PREV_HASH` for the first append)
3. Computes `hash = HMAC-SHA-256(canonicalForm(event))`
4. Writes the JSON line + '\n', `fsync`s, closes

Appends are gated by a single in-process promise chain
(`private chain: Promise<unknown> = Promise.resolve()`) — two
concurrent `await writer.append(...)` calls serialize. This is
deliberately single-process: the writer does not coordinate
across processes (a v0.4 follow-up is `flock(LOCK_EX)` for
multi-process correctness, but the current threat model is "one
TUI session at a time per user").

### 4. The chokepoint decorator: `withAuditChokepoint`

`withAuditChokepoint(inner, sink)` wraps a `Chokepoint` and
records every decision the inner chokepoint makes. The wrapper
itself is the only place that calls `sink.append('chokepoint-decision', ...)` — the inner chokepoint is unchanged. This
keeps ADR-0006's promise (the chokepoint is the only hard gate)
intact: auditing is a *post-decision* observation, never a
co-routine that could refuse a call the inner chokepoint
allowed.

`AuditSink` is the indirection that lets tests assert on what
was logged without touching the file. The production default
is `makeAuditSink({ writer })`; `NOOP_SINK` is the opt-out for
benchmarks and the `GMFT_AUDIT=off` env var.

### 5. The CLI: `gmft audit {verify,log,tail}`

Three subcommands, all reading the default audit path:

- `verify` — walk the chain, recompute every hash, exit 0 if
  intact, 1 if broken. Prints the broken line number, the
  recorded vs. computed hash, and the count of events verified
  before the break.
- `log` — read with filters (`--limit N`, `--since ISO`,
  `--until ISO`, `--kind KIND`...); prints
  `<line>\t<ts>\t<kind>\t<JSON-encoded payload>`.
- `tail` — follow the file, emit each new line, stop on SIGINT.
  Uses a 500ms `statSync` + read-poll (deliberately not
  `fs.watch`, which is platform-flaky in tests and the cost of
  one poll/sec is invisible at human-eye timescales).

The dispatch is an early branch in `cli.tsx` (right after the
`meow` block, before onboarding or TUI mount) — `gmft audit
...` is a subcommand, not a flag, and it should never trigger
onboarding.

### 6. Opt-out: `GMFT_AUDIT=off`

The audit log is *always on by default*. `GMFT_AUDIT=off`
swaps the sink for `NOOP_SINK` in `makeAuditSink`. The env var
is read at the binary boundary (in `cli.tsx`) and threaded
into the writer, not at the sink call site — so the opt-out
decision is one grep away in the code. There is no "log
nothing for this session only" path: the audit log is a
*system* invariant, and opt-outs are environmental (test
fixtures, benchmarks), not per-session.

### 7. The TUI is a deliberate non-change in v0.3.C

`AgentApp.tsx` does **not** call `withAuditChokepoint` in
v0.3.C. The plan budget for v0.3.C is the audit library + CLI;
the TUI wiring is a follow-up commit that will land in v0.3.D
or v0.3.E. A 6-line breadcrumb comment in `AgentApp.tsx`
points at this ADR so the next reader knows the gap is
intentional, not forgotten.

**Why ship the library first?** The CLI (`gmft audit verify`)
is the *interesting* surface — it answers a question no UI
element can. Shipping the CLI lets an external CI / cron job
verify the log without an interactive session. The TUI
follow-up adds the in-session view but doesn't change what
the log captures.

## Consequences

- The audit log is an additional disk write on every
  chokepoint decision. The writer is fsync-on-append, so a
  10-event/min TUI session adds ~10 fsyncs/min to the user's
  SSD. This is within tolerance (consumer SSDs handle
  10^5+ fsyncs/day without measurable wear) but is a
  parameter the v0.4 follow-up can expose (e.g.
  `GMFT_AUDIT_FSYNC=off` for the dev-loop case).
- The chain is *not* cross-process safe. Two TUI sessions on
  the same `$XDG_CONFIG_HOME` would race. The v0.3.C scope is
  one TUI per user, so this is a known limitation, not a bug.
- The audit log is *evidence*, not *enforcement*. It does not
  stop a chokepoint decision from happening — it records what
  happened. A future ADR may add a "block on log write failure"
  policy, but v0.3.C treats writer errors as warnings (logged
  to stderr) so a full disk doesn't take down the TUI.
