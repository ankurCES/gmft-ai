# ADR-0016 — Supervisor fires land in the audit log

- Status: Accepted
- Date: 2026-06-18
- Plan: v0.4-A §A.3 (Supervisor state in the audit log)
- Builds on: ADR-0013 (audit chain HMAC), ADR-0014 (supervisor completion)

## Context

The v0.3.C audit log records chokepoint decisions, tool calls, and
session lifecycle events in a tamper-evident HMAC chain (see
ADR-0013). v0.4-A added the supervisor (ADR-0014): a wrapper around
`runTurn` that observes the agent's tool-call stream and emits
`supervisor-fire` events for rule violations (loop-detected,
overclaim, plan-issue, risk-escalation).

The supervisor's fires were previously only visible in the in-memory
session log (via `lastFires()` / `lastPostmortem()` on the
`withSupervisor` wrapper). They were NOT in the audit log, which
meant:

1. Post-session review in `gmft audit log` couldn't see the supervisor's
   fires — the reviewer had to cross-reference the session log.
2. The hash chain didn't cover supervisor decisions, so an attacker
   who compromised the session log could not be cross-checked against
   the audit chain.
3. The verifier (`gmft audit verify`) couldn't prove supervisor fires
   weren't tampered with.

## Decision

Add a new audit event kind, `supervisor-fire`, plus a new decorator,
`withAuditSupervisor`, that mirrors `withAuditChokepoint` at the
iterable layer rather than the object layer.

### Why the iterable layer

`withAuditChokepoint` wraps a `Chokepoint` (an object with a `decide`
method). The supervisor is structurally different: it's a transform
over an `AsyncIterable<AgentEvent>` (the output of `runTurn`). The
natural instrumentation point is to wrap the iterable and watch each
yielded event for `type === 'supervisor-fire'`. The decorator returns
a new iterable that yields the same events unchanged, but calls
`sink.append('supervisor-fire', payload)` fire-and-forget for each
supervisor fire.

### Why not wrap `withSupervisor` itself

We could have added the audit call directly to `withSupervisor`'s
body (where each fire is already constructed). Two reasons against:

1. **Separation of concerns.** `withSupervisor` is a pure transform
   that adds advice to the LLM's history; audit emission is a
   side-effect that should be opt-in via the same decorator pattern
   the rest of the codebase uses.
2. **Consistency with `withAuditChokepoint`.** The chokepoint wrapper
   doesn't emit audit events directly; `withAuditChokepoint` does.
   Matching that pattern means the wiring story for downstream
   consumers is uniform.

### Payload shape

The audit payload is a `Record<string, unknown>` built by spreading
the fire first (so variant-specific fields land at the top level),
then explicitly setting the truly common fields (`kind`, `advice`,
`targetEventId`). The `severity` and `text` fields are NOT
explicitly set — they exist only on `PlanIssueFire` and would be
`undefined` (and thus absent) on other variants.

| Variant              | Common (always)    | Variant-specific fields               |
| -------------------- | ------------------ | ------------------------------------- |
| `LoopDetectedFire`   | `kind`, `advice`,  | `tool`, `count`, `recent`             |
|                      | `targetEventId`    |                                       |
| `OverclaimFire`      | `kind`, `advice`,  | `quote`, `evidence`                   |
|                      | `targetEventId`    |                                       |
| `PlanIssueFire`      | `kind`, `advice`,  | `severity`, `text`                    |
|                      | `targetEventId`    |                                       |
| `RiskEscalationFire` | `kind`, `advice`,  | `tool`, `firstToolOfTurn: true`       |
|                      | `targetEventId`    |                                       |

Consumers of the audit log should narrow on `kind` first to know
which additional fields are present.

### Fire-and-forget semantics

Same rationale as `withAuditChokepoint`: the agent loop yields
`tool-call-request` synchronously off `runTurn`'s output. Blocking
on the audit append would couple supervisor-fire audit latency to
tool-call latency, which would degrade the user's experience. The
hash chain doesn't need strict serialization with the supervisor's
yield — it needs the events to land in order, and the writer's
mutex (Task 2 of ADR-0013) guarantees that.

### Downstream wiring (AgentApp)

The plan calls for `AgentApp` to wire the chain as
`chokepoint → withAuditChokepoint → withSupervisor → withAuditSupervisor`.
`withSupervisor` must run before `withAuditSupervisor` because the
audit decorator reads its output events. AgentApp lives in a
downstream package that doesn't yet exist in this monorepo; the
wiring will land there. The two decorators (`withAuditChokepoint`
and `withAuditSupervisor`) are exported from `@gmft/core`'s
`audit/instrument.js` so AgentApp can compose them with one import.

## Consequences

- Post-session review (`gmft audit log --kind supervisor-fire`) now
  shows every supervisor fire, in order, with the same hash-chain
  guarantee as chokepoint decisions.
- The verifier can prove supervisor fires weren't tampered with
  alongside the rest of the audit chain.
- Two decorators to wire in AgentApp (4 lines of glue); no other
  downstream changes needed.
- The audit log now grows by 1 event per supervisor fire. Fires are
  rare in practice (rule-engine only fires on violations) so this is
  not a size concern.

## Alternatives considered

1. **Inline audit emission in `withSupervisor`.** Rejected: breaks
   separation of concerns, makes audit mandatory, and diverges from
   the `withAuditChokepoint` pattern.
2. **Add supervisor fires to the existing `chokepoint-decision` kind.**
   Rejected: the two are semantically distinct (chokepoint = binary
   allow/deny/confirm gate; supervisor = rule-violation advice).
   Conflating them would make the audit log harder to filter.
3. **Make `withAuditSupervisor` return an augmented `lastFires()`
   accessor.** Rejected: the audit log is the source of truth for
   post-session review; ad-hoc accessor APIs would drift.
