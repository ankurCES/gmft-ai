# gmft-ai v0.4 plan — supervisor completion + AD attack tools

**Date:** 2026-06-18
**Status:** Draft (awaiting review)
**Target release:** v0.4.0 (~+150 tests, ~3 weeks)
**Prior release:** [v0.3.0](../CHANGELOG.md#030--2026-06-19) — 855 tests green

## 1. Goal

Two sub-phases, each individually mergeable, each with its own ADR:

- **v0.4-A — Supervisor completion.** The supervisor was *introduced* in
  v0.2.A (rules A/B/C + postmortem) and partially surfaced in v0.3.A
  (StatusRail, fire markers). v0.4-A *completes* it: adds the two
  remaining triggers from [ADR-0009](../plans/adr/0009-multi-agent-supervisor.md)
  (3: overreach, 5: risk escalation), replaces the rule-only plan-quality
  trigger with an LLM-judge hybrid (rule fast-paths, LLM judge on
  ambiguous cases), and wires supervisor state into the audit log so
  every fire is tamper-evident next to the chokepoint decision.
- **v0.4-B — AD attack tools.** Adds 5 Active Directory tools behind
  the same hard gate wifi's evil-twin uses (`requiresElevation +
  destructive + typeToConfirm`). New `gmft/ad:0.4` Docker image. Closes
  the "no AD category" gap that the v0.3.B catalog doc flagged.

Both sub-phases are independent. v0.4-A can ship without v0.4-B and
vice versa. v0.4.0 is the aggregate tag; per-sub-phase tags land
along the way.

## 3. v0.4-A — Supervisor completion

### A.1 — Trigger 3 (overreach) + Trigger 5 (risk escalation)

ADR-0009 explicitly defers these to "a later release" because they
needed more thought than v0.2.A had time for. v0.4-A is that release.

**Trigger 3 (overreach):** the agent invokes a destructive tool whose
`args.target` is not the session target (or --scope allowlist) but is
still inside the chokepoint's denylist range — i.e. the call would be
allowed by the denylist but is morally out-of-scope. Existing chokepoint
already flags this in the StatusRail destructive-warning surface
(v0.3.B); v0.4-A promotes it to a supervisor fire.

**Trigger 5 (risk escalation):** the agent went from read-only to
destructive tools within a single turn without any intervening recon.
The supervisor reads the turn's `tool-call-request` events and fires
when the first destructive call has no prior recon-class call in the
same turn.

Files:
- `packages/core/src/agent/supervisor-rules.ts` — add rules 3 + 5
- `packages/core/test/supervisor-rules-trigger-3.test.ts` — 3 tests
- `packages/core/test/supervisor-rules-trigger-5.test.ts` — 3 tests

### A.2 — LLM-judge for plan quality (replaces rule-only)

Today's Rule A ("destructive without recon") is a pure rule. It
fires on `nmap → nikto` but misses `nmap → whatweb → sqlmap` where
the recon was technically present but insufficient. v0.4-A adds an
LLM judge that runs on ambiguous cases (rule did NOT fire but the
turn looks suspicious).

Design:
- After Rule A's rule-only pass, if the agent's first destructive
  call in the turn is `sqlmap` / `nuclei` (the high-blast-radius
  attack tools), the supervisor fires ONE additional LLM call:
  "given this turn's tool calls + a 1-paragraph summary of the
  findings so far, was the agent's pre-attack recon sufficient?"
- The LLM's verdict (`sufficient | insufficient | unclear`) becomes
  a `SupervisorFire` of a new kind: `plan-issue-llm`. The chat
  transcript gets a fire marker the same way rule-only fires do.
- Cost: bounded — at most 1 extra LLM call per turn, and only when
  a high-blast-radius tool is called. Most turns (recon, reports,
  read-only) never trigger it.

Files:
- `packages/core/src/agent/supervisor-rules.ts` — add the
  ambiguous-case dispatch
- `packages/core/src/agent/supervisor-judge.ts` — new file, the LLM
  judge wrapper (interface + provider-agnostic; uses the same
  `LanguageModel` the agent uses, falls back to NOOP if the user
  passes `--no-supervisor-judge`)
- `packages/core/test/supervisor-judge.test.ts` — 4 tests (sufficient,
  insufficient, unclear, NOOP fallback)

### A.3 — Supervisor state in the audit log

a second audit event kind, `supervisor-fire`, that records every
supervisor fire with the same `prevHash → hash` chain. This makes
supervisor state tamper-evident and visible in `gmft audit log --kind supervisor-fire`.

- `packages/core/src/audit/sink.ts` — no change (the sink is generic)
- `packages/core/src/agent/instrument.ts` — new file, the

## 4. v0.4-B — AD attack tools

### B.1 — Threat model (read this first)

AD attack tooling is the single most dangerous category gmft will
ship. The chokepoint must enforce the same hard gate the wifi
evil-twin already uses, plus two AD-specific constraints:

1. **Same gate as evil-twin:**
   - `requiresElevation: true` → env opt-in `GMFT_ALLOW_ELEVATION=true`
   - `destructive: true` → y/n confirm
   - `typeToConfirm: 'ATTACK'` → user types the literal word

2. **AD-specific:**
   - **No `--scope` fan-out for AD tools.** `--scope` (v0.3.B) is
     designed for recon, where running nmap against 50 hosts is fine.
     AD attack tooling on 50 hosts in parallel = 50 simultaneous
     attempts at lateral movement = detection + containment. The
     AD tools MUST be called one target at a time. The chokepoint
     should reject `--scope` for any tool flagged `category: 'ad'`.
   - **Domain controller is always blocked.** The chokepoint should
     refuse any AD tool call where `args.target` matches the session
     machine's own domain controller (resolved via `realm list` /
     `klist` if the user opts in via `GMFT_REALM_LOOKUP=true`).
     Operators running gmft on a workstation should not be able to
     spray-credential the DC without explicit scope.
   - **Credential exposure in transcript.** `secretsdump` outputs
     NTLM hashes. The session log + audit log + reports all need
     redaction for `sam-account-name:hash` patterns. v0.4-B adds a
     `redactAdSecrets` pass that runs at transcript-write time
     (mirror of the v0.1.5g `redactSecrets` work but tuned for
     impacket output).

These three constraints are documented in `docs/safety.md` (the
existing chokepoint threat-model doc) under a new §6 "AD-specific
threat model" before any AD tool code lands.

## 7. Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Impacket output changes between versions break parsers | medium | medium | Pin `impacket==0.11.0` in Dockerfile; parse the most stable output shapes (table rows, JSON dumps) and have fallbacks |
| LLM judge (A.2) gives a noisy verdict and floods the chat with fires | medium | medium | Default `--supervisor-judge off`; only opt in via `GMFT_SUPERVISOR_JUDGE=true`; rate-limit at 1 judge call per turn |
| AD tools' docker image grows large (impacket + bloodhound + their deps) | low | low | Use `--no-cache-dir` and `slim` base; multi-stage if it gets > 500 MB |
| `--scope` ban for AD tools breaks an operator's workflow | low | medium | Ban is opt-out via `--allow-ad-scope` flag (not env var — flag is more discoverable for a one-shot decision) |
| DC-block rule is too aggressive (false positives) | medium | medium | Rule fires on `realm list` output, not hostname guessing; if `GMFT_REALM_LOOKUP=false` the rule is skipped (operator knows their DC) |
| Transcript redaction misses a new impacket output format | medium | high | The redaction is regex-based with explicit test fixtures; the plan budget includes 3 tests but the function is small enough that more can land if reports show gaps |

## 8. Phase ordering

v0.4-A and v0.4-B are independent. Recommendation: **land A.1-A.4
first** (smaller blast radius, completes a deferred ADR), **then
B.1-B.7** (new tools, new image, new gate semantics — bigger review
surface).

Rationale: A.1-A.4 is the completion of a feature the user asked for
back in v0.2; it's a smaller review and unblocks the supervisor CLI
which B.5's redaction work might want to call into. B.1-B.7 then
lands on a stable supervisor base.

If B is more urgent (e.g. a customer needs AD tools for an upcoming
engagement), swap the order — the only constraint is "both merge to
main before the v0.4.0 tag".
