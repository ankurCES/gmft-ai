# GMFT-AI вҖ” Implementation Plan

> **Good-Made-for-the-World AI** вҖ” a TUI-first, agentic penetration-testing runtime.
> Chat is the surface. Safety is the floor. Defenders, bug-bounty hunters, and red-teamers
> on systems they own or have explicit written permission to test.

| | |
|---|---|
| **Repo** | `github.com/ankurCES/gmft-ai` (public, MIT) |
| **Target v0.1** | 6 phases, 12 weeks, ~50 tests, single `pnpm install && pnpm build && pnpm test` workflow |
| **Primary surface** | Interactive Ink/React TUI chat вҖ” no web UI in v0.1 |
| **Reference repos** | [fluxionnetwork/fluxion](https://github.com/fluxionnetwork/fluxion), [vxcontrol/pentagi](https://github.com/vxcontrol/pentagi), [0x4m4/hexstrike-ai](https://github.com/0x4m4/hexstrike-ai), [xalgord/xalgorix](https://github.com/xalgord/xalgorix) |
| **Inspirations, not copies** | fluxion = workflow shape, pentagi = multi-agent + sandbox pattern, hexstrike-ai = tool catalog shape, xalgorix = UX/observability shape |

## 0. What this document is

A **task-by-task** plan. Every phase is broken into commits (where useful) and numbered tasks.
Each task is small enough to be a single PR / single agent turn: an isolated change, a test,
a commit. The order is strict вҖ” later tasks depend on earlier ones вҖ” but tasks within the
same phase can sometimes be parallelised.

The plan deliberately defers:

- **Multi-agent supervisor** (pentagi-style adviser/monitor) вҶ’ v0.2
- **Knowledge graph memory** (Graphiti/Neo4j) вҶ’ v0.2 (we ship flat JSONL + TF-IDF in v0.1)
- **Binary exploitation tools** (gdb, pwntools, ghidra) вҶ’ v0.2
- **Cloud-native deployment** (k8s, multi-tenant) вҶ’ v0.3
- **Tauri desktop wrapper** вҶ’ out of scope (TUI is the product)
- **MSP/team mode** вҶ’ v1.0

These are called out where they are deferred so we can design v0.1 cleanly without
re-architecting later.

**Key design decisions** (and why):

1. **TUI first, no web UI.** Ink is React; we get composable components, state hooks,
   streaming by rerendering on a delta buffer. TUI is a peer of, not a lesser than, the
   web вҖ” pentesters live in terminals.

2. **Hand-rolled ReAct loop, not LangChain.** LLM streaming + tool calling is well-served
   by Vercel AI SDK (`ai` package). We compose a tight 80-line loop around it that knows
   about our tool registry, chokepoint, and memory. Avoids LangChain's 200-MB dep tree and
   bad TUI fit (sync-only tool calls, opaque retries).

3. **Chokepoint = single safety gate.** Every tool call вҖ” read or write вҖ” passes through
   `Chokepoint.confirm(ctx, tool, args) вҶ’ Decision { Allow, Confirm(reason), Deny(reason) }`.
   This is the moral equivalent of pentagi's container isolation, expressed in code we can
   test. It's the **floor** of the system, the thing we unit-test most aggressively.

4. **Tools are typed.** Each tool is a `Tool<Input, Output>` Zod-schema'd module in
   `src/tools/<category>/<name>.ts`. The registry is generated at build time вҶ’ TUI side
   rail auto-updates. No magic strings.

5. **Sandbox by default, host-fallback with explicit warning.** Pentagi's whole point is
   isolation. We do the same: prefer Docker (configurable image), fall back to host only
   with a loud persistent banner ("вҡ  HOST-LEVEL EXECUTION вҖ” actions are NOT isolated")
   that the user must acknowledge. AppArmor/landlock profiles are stretch (v0.2).

6. **Flat JSONL session log, not a DB.** Every turn: `{ts, role, content, tool_calls,
   tool_results, decision, tokens_in, tokens_out}` appended to `~/.local/share/gmft/sessions/<id>.jsonl`.
   v0.1 doesn't need pgvector. v0.2 can layer a vector index over the same files.

7. **Multi-provider LLM via Vercel AI SDK.** OpenAI, Anthropic, Google, OpenRouter, Ollama,
   and any OpenAI-compatible endpoint. Provider selected by `~/.config/gmft/config.toml`.

## 6. Safety & threat model (the one-paragraph version)

GMFT-AI is a *powerful* tool. The chokepoint is the single point through which all tool
calls flow. Default policy: read-only tools allowed, destructive tools require typed
confirmation per call, elevated tools require an env-var opt-in, and **no tool ever runs
without an explicit target that the user has named in the session or on the CLI**. Targets
matching RFC1918, link-local, localhost, or `.internal` are denied unless
`GMFT_ALLOW_PRIVATE=true`. Every tool invocation is logged to the append-only session log
with a content hash so it can be replayed. We do **not** attempt to stop a determined
adversary with root on the host; we attempt to make the default path safe enough that an
analyst who is having a bad day cannot accidentally nuke production. Full threat model
lives in `docs/safety.md` (written in phase 6).

## 8. Phase-by-phase acceptance criteria

A phase is **done** when:

1. All its tasks are merged to `main`.
2. `pnpm -r test` is green.
3. `pnpm -r build` produces a runnable `apps/gmft/dist/cli.js`.
4. The CHANGELOG entry is written.
5. A demo cast (or a new screenshot) lands in `docs/screenshots/`.
6. A `v0.1.0-phaseN` git tag is pushed.
7. A short PR description exists linking to this plan section.

A `v0.1.0` release requires **all 6 phases** plus:

- A signed release artifact (`pnpm pack` вҶ’ tarball, sha256 sum).
- README quickstart works on a fresh Ubuntu 24.04 VM with `node 20`, `pnpm`, and Docker
  (in that order) вҖ” verified by the release checklist in `docs/release-checklist.md`.
- `docs/safety.md`, `docs/threat-model.md`, `docs/tool-catalog.md` exist and are linked
  from README.

## 9. Risks & mitigations

| Risk | Mitigation |
|---|---|
| Ink/React 19 still settling | Pin versions tightly, ship a `<TuiBoundary>` error boundary so a crash in one component doesn't kill the TUI |
| Vercel AI SDK API churn | Pin to a minor version, integration-test against the pinned shape |
| Tool false-confidence вҖ” agent thinks `nmap` is a complete scan | System prompt is explicit; Findings tab exposes the gap; v0.2 supervisor catches it |
| Live tool tests are slow and flaky in CI | Live tests are opt-in only; CI uses fixture-replay; flake-watch workflow |
| LLM emits a destructive tool call without confirm | Chokepoint is the single gate; tested with 12 rule cases; CI matrix on policy changes |
| User pastes API key into chat | Provider keys live in env, never the session log; `session/log.ts` redacts `api_key`, `token`, `password` substrings |
| Wifi evil-twin tool run on wrong AP | Three layers of friction: (a) `requiresElevation` env opt-in, (b) `destructive: true` confirm, (c) `attack` typed confirm, (d) status rail shows target BSSID throughout |

## 11. Execution

This plan is the contract. Phases are sequential. Tasks within a phase are
sequential by default; anything marked `[parallelizable]` in a future revision can be
fanned out.

We will commit per-task (small, reviewable), and at the end of each phase we
**verify the phase acceptance criteria above** before tagging.

We do **not** start phase N+1 until phase N is tagged. We do **not** tag
"v0.1.0" until all 6 phases meet the v0.1.0 release checklist.

The "make the world a safer and better place" framing is real, not marketing: every
phase has a safety task, every tool has a chokepoint test, every release has a
threat-model doc, and the README leads with the legal disclaimer in the same
size as the title.
