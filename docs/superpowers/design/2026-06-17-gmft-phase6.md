## What this is

Phase 6 is a 4-feature delta on top of phases 1-5. This design doc
exists to answer the question "are these the right features, in the
right order, with the right architectural seams?" before we start
writing code. The implementation plan answers "what code do I write
first?" once the design is approved.

Read the spec for the full per-feature detail. This doc is the
distilled design summary + the questions that actually need a human
decision.

## §1 — The 4 features

| # | Feature | Risk | Tests | Order |
|---|---------|------|-------|-------|
| C | `wifi_deauth` + `wifite_scan` | Low (additive tools) | +6 | **1st** |
| B | `report_write` + FindingsTab rewrite | Medium (new file format, new sidecar) | +7 | 2nd |
| A | `attack_chain` tool + `runInner` seam | **High (architectural)** | +13+ | 3rd |
| D | `targetsFromFile` + `executeWithScope` | Medium (consumes the A seam) | +5 | 4th |

Cumulative: 288 → ~319 tests across the 4 features.

## §2 — The order: C → B → A → D

**Why C first:** lowest risk, pure additive, no architectural changes.
Validates the "add a tool with a new Dockerfile" pattern. ~300 LOC,
~30 min of focused work.

**Why B second:** B introduces a new file format (the report) and a new
sidecar (`{sessionId}.selections.json`) but no new *type* of interaction.
The FindingsTab rewrite is UI-only. B is "more surface area" without
"new architecture." ~400 LOC, ~1 hour.

**Why A third, and why it's the centerpiece:** A introduces the
`runInner` seam — the public function macro tools use to invoke other
tools from inside their own `run`. This is the architectural decision
that unlocks chains, repeat-with-backoff, best-of-n, and parallel
fan-out (the post-v0.1 tools). It also introduces 4 new `AgentEvent`
variants and a new UI component. ~600 LOC, 3-4 hours.

**Why D fourth:** D *consumes* the `runInner` seam (scope-file expansion
calls `runInner` per target). Building A first means D is just "add a
file reader + an `executeWithScope` method that loops." ~250 LOC, 1-2
hours.

The alternative (A before B) puts the architectural change in the
middle of the feature batch, which makes it harder to bisect if
something regresses. C → B → A → D means the first half is purely
additive and the second half is a single architectural change + a
single consumer of that change. Cleaner bisection.

## §3 — Feature A: the architectural centerpiece

The whole of phase 6 hangs on the `runInner` seam. The design question
is: **how do we let one tool invoke another tool's logic from inside
its own `run`, while preserving the chokepoint's deny-a-single-step
property?**

### The chosen design (3 layers)

```
┌─────────────────────────────────────────────┐
│  Agent loop (runTurn)                       │  ← existing, unchanged
│    → execute(tool, args, opts)              │
└─────────────────┬───────────────────────────┘
                  │ (wraps)
                  ▼
┌─────────────────────────────────────────────┐
│  runInner(tool, args, registry, chokepoint, │  ← NEW (30 LOC)
│           ctx, opts)                        │
│  - re-uses chokepoint rules                 │
│  - re-uses findings.jsonl append            │
│  - opts.suppressTypeToConfirm for chains    │
└─────────────────┬───────────────────────────┘
                  │ (called by)
                  ▼
┌─────────────────────────────────────────────┐
│  attack_chain.run(args, ctx)                │  ← NEW
│  - ctx.innerRunner = runInner curried with  │
│    registry/chokepoint/ctx from outer call  │
│  - sequential per-step loop                 │
│  - emits 4 new AgentEvents                  │
└─────────────────────────────────────────────┘
```

**Why a public function, not a class method on the executor:** the
chain tool lives in `@gmft/tools`, the executor lives in `@gmft/core`.
A public function in `@gmft/core` exposed via a `ToolContext` field
keeps the dependency direction clean (`@gmft/tools` doesn't import
`@gmft/core`'s internals beyond the public types). Macro tools
(repeat-with-backoff, best-of-n) plug into the same hook.

**Why `suppressTypeToConfirm: true` is an opt-in flag, not the
default:** the chain's outer `typeToConfirm: 'attack'` covers the
chain; the per-step `typeToConfirm: 'scan'` (e.g. for an `nmap`
step) would otherwise prompt the operator a second time for each
step. The flag is passed by the chain tool, not by the chain's
caller. Default is `false` so future macro tools that DON'T want
suppression get it for free.

**Why `execute()` becomes a thin wrapper around `runInner`:** it
keeps the chokepoint logic in exactly one place. The chain tool
gets the same chokepoint treatment as the agent loop, by construction.
The refactor is ~30 LOC, fully testable, and the existing 301 tests
guard the regression.

### The chain audit log format (spec §9 #1, resolved)

**One audit entry per chain, with nested `steps: [...]`.**

```json
{
  "ts": "2026-06-17T15:30:00Z",
  "action": "attack_chain",
  "chainId": "uuid",
  "totals": { "total": 3, "completed": 2, "denied": 0, "erred": 0, "skipped": 1 },
  "steps": [
    { "stepIndex": 0, "tool": "nmap", "args": {...}, "status": "ok", "findingIds": ["f-1", "f-2"], "durationMs": 1234 },
    { "stepIndex": 1, "tool": "nikto", "args": {...}, "status": "ok", "findingIds": ["f-3"], "durationMs": 5678 },
    { "stepIndex": 2, "tool": "metasploit", "args": {...}, "status": "skipped", "reason": "previous step failed", "durationMs": 0 }
  ]
}
```

**Why:** the session's `findings.jsonl` contract is "one line per
finding" and that's preserved. The audit log gets a richer entry
for chains because the chain *is* the action — the steps are
detail. The chain's `chainId` is the cross-reference key (matches
the `AgentEvent.chainId` field).

**Rejected alternative:** one JSONL line per step. This would
fragment the chain across 4+ audit lines, making "show me what
chains ran" queries a 4-way join. The single-entry-with-nested-steps
is queryable in one read.

## §4 — Feature D: scope files

The `targetsFromFile` flag is additive on `Tool`. The chokepoint's
contract doesn't change. The scope-file expansion happens inside
`executeWithScope`, which calls `runInner` per target (consuming
Feature A's seam).

### Why the seam lets D be small

Without `runInner`, D would have to either:
- Re-implement the chokepoint logic in `executeWithScope` (duplication,
  drift risk), or
- Call `execute()` in a loop (which would re-prompt for elevation
  per target — wrong)

With `runInner`, D is: read file → for each line, clone args →
call `runInner` → accumulate. ~50 LOC. The chokepoint rules fire
per-target as designed.

### The chokepoint sees the file path, not the line

This is the one subtle decision. The outer `executeWithScope` call
passes the **file path** as `args.target` for the chokepoint's
"is this a private network / on the denylist" check. The chokepoint
doesn't need to be smart about file paths; it just applies the same
rules to a string. If the file path is on a denylist (e.g.
`/etc/shadow`), the whole scope-file call is denied before any
target runs.

The per-target lines go through the chokepoint too (via
`runInner` → `checkTarget`), so a file with 1000 lines where 5 are
denylisted produces "5 denied, 995 ok" — the operator gets
visibility into which targets were skipped.

### argv substitution (spec §9 #3, resolved)

**Clone `baseArgs`, replace `target` field per line.** Other args
stay constant.

```ts
// executor.ts
for (const line of lines) {
  const perTargetArgs = { ...baseArgs, target: line };
  const result = await runInner(tool, perTargetArgs, registry, chokepoint, ctx, opts);
  results.push(result);
}
```

**Why not token substitution (`{target}` in argv strings):** the
tool's args are a structured object (`Record<string, unknown>`),
not a string template. Token substitution would require a parsing
layer and would lose zod validation. Clone-and-replace is one line
and trivially correct.

**Rejected alternative:** mutate `baseArgs` in place. Bad — the
next iteration would see the previous line's target.

## §5 — Feature B: report writer

### The output path policy (spec §9 #2, resolved)

**Default:** `~/.local/share/gmft/reports/{sessionId}.{ext}`.

**Override:** `outputPath?: string` arg. Normalized via `path.resolve`,
verified to be inside the reports dir (no `..` escape) — `..` escape
throws a tool error. If the file already exists, append
`.{timestamp}.` before the extension.

**Why not allow any path:** the report is the only output a pentest
engagement keeps long-term. Constraining it to the reports dir makes
the audit story cleaner ("all reports live in X") and prevents the
operator from accidentally writing to a sensitive location.

### The selections sidecar

`{findingsDir}/{sessionId}.selections.json` — `{ checkedIds: string[] }`.
Written by the TUI on checkbox change (debounced 500ms). Read by
`report_write` to filter findings: if `checkedIds.length > 0`, include
ONLY those ids; otherwise fall back to the severity filter.

**Why a sidecar, not in `findings.jsonl`:** the selections are
ephemeral TUI state, not findings. Mixing them into `findings.jsonl`
would confuse downstream tools (which expect one finding per line
and would try to render the selections as a finding).

## §6 — Things explicitly out of scope for phase 6

These are deferred to v0.2+ with their own design docs:

- **PDF reports** (B2) — `wkhtmltopdf` or `puppeteer` post-process of
  the HTML report. Not a design problem, just an integration task.
- **Parallel fan-out across scope-file targets** (D2) — would need
  concurrency limits, error aggregation across goroutines, and a
  per-target progress UI. Phase 6 is sequential by design.
- **Persisted chain state across sessions** — chains live and die in
  one session. Resuming a chain after a crash is a v0.2 problem.
- **Multiple parallel chains** — the LLM can only call one tool at a
  time. If two chains need to run concurrently, that's a v0.2
  problem (probably needs a queue).
- **Top-level `scope: string[]` on `ChokepointCall`** (D2 from spec) —
  cleaner long-term but a breaking change to the chokepoint's
  contract. Deferred to v0.2 with an ADR.

## §7 — Risks (carried from spec §8, with task-level mitigations)

The implementation plan (§13) maps each spec risk to the specific task
that mitigates it. The summary:

| Risk | Mitigation |
|------|------------|
| `runInner` becomes a god-object | The seam is documented + only 2 callers in phase 6 (chains, scope-files). The type system enforces the contract. |
| Chain suppresses a prompt the operator wanted | Integration test in A.3 asserts suppression is scoped to the chain's outer type. |
| Scope file is huge | 10k line + 1MB caps throw before execution. |
| FindingsTab rewrite breaks existing test | New test file replaces the placeholder. |
| Report writer is slow | O(n) in finding count; sub-100ms for 1000 findings. |
| LLM tricked into 20-step chain | `steps.max(20)` zod cap; chain is operator-confirmed end-to-end; audit log records the full step list before step 1 runs. |

## §8 — Open questions for human review

The 3 spec §9 questions are resolved (see §3, §4, §5 above). The
remaining open questions for the human reviewer:

1. **Is the `runInner` seam the right level of abstraction?** The
   seam is a function, not a class. It takes the registry +
   chokepoint + ctx as explicit args. Alternative: a `MacroContext`
   object that bundles them, plus per-call opts. The function form
   is simpler and more testable; the object form would be more
   extensible for future macro tools that need MORE than what the
   outer call provides (e.g. a separate audit log per chain step).

   **My recommendation:** the function form. If a future macro
   tool needs more, add a second arg, not a redesign.

2. **Should the chokepoint's outer check on a scope file be
   "is the file path on a denylist" or "is the file readable by
   the operator"?** Currently the spec is denylist only. The
   "readable" check is a permission question (chokepoint doesn't
   know about filesystem permissions, only the chokepoint's own
   denylist).

   **My recommendation:** denylist only for phase 6. Filesystem
   permissions are a deployment concern (AppArmor / user-systemd),
   not a chokepoint concern.

3. **Is `~/.local/share/gmft/reports/` the right default location?**
   Other tools in the project use `~/.local/share/gmft/` for state
   (verifiable by `find packages -name 'paths.ts'` or similar).
   If the existing convention is different, this should match.

   **My recommendation:** verify against the existing code; if no
   convention exists, `~/.local/share/gmft/reports/` is the XDG
   default and is fine.

## §9 — Approval

If these 4 features, in this order, with this architectural seam, look
right, approve and I'll start executing the implementation plan
feature-by-feature. If you want to:

- **Reorder** (e.g. A before B): possible, the seam is the same. The
  implementation plan's task numbers are arbitrary; renumber as
  needed.
- **Add a feature:** check the test budget impact first. Anything
  beyond +10 tests should be its own phase.
- **Remove a feature:** A is the architectural centerpiece and
  unlocks the most future work. C is the lowest risk and the best
  warmup. B and D are medium-risk and could each be cut without
  blocking the other 3.
- **Change the seam design:** this is the highest-leverage decision
  in the phase. If you want a class method, a MacroContext object,
  or a different shape, now is the time.
