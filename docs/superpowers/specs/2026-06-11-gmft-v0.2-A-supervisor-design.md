# v0.2.A — Multi-agent supervisor (pentagi-style)

> First sub-project of v0.2. Brings the v0.1 master plan's §0 and §7
> deferral ("Multi-agent supervisor (pentagi-style) — `agent/loop.ts` is
> a single file with clear seams for `Mentor.interveneIfStuck()` — v0.2
> plugs in") into a shippable design against the current codebase
> (main at `bcc3bb4`, post-v0.1.0 release).

**Goal:** A strictly-silent second observer that watches the agent
loop's `AsyncIterable<AgentEvent>`, fires rule-based observations on
plan quality / loops / confidence, and writes one LLM-generated
postmortem at end of turn. The supervisor **cannot** block tool calls
and **cannot** prompt the user. The chokepoint stays the only hard
gate.

**Why this exists:** v0.1's chokepoint enforces policy (deny destructive
calls outside scope, require confirm for destructive, require typed
confirm for high-friction). It does **not** catch a *loop* (same call
4× in a row), an *overclaim* (agent says "scan complete" with no
findings), or a *plan-quality* issue (agent goes destructive without
recon). The v0.1 risk list §9 calls two of these out explicitly
("tool false-confidence"; "interveneIfStuck" in the deferral table).
This feature closes those gaps.

**What this is NOT:**
- Not an LLM-in-the-loop that rewrites the agent's prompts every step.
- Not a deny gate (chokepoint is the deny gate).
- Not a user prompt (no "supervisor wants your attention" dialogs).
- Not a vector/graph memory (that's v0.2.B+E).
- Not a binary-exploitation-tool feature (that's v0.2.C).

**Test budget:** 46 new tests. v0.1 baseline 374 → v0.2.A total ~420.
Build/typecheck stays clean. ~0 new runtime dependencies.

## 1. Scope

The supervisor fires on four triggers. Two are explicit in the v0.1
plan; two are derived from the pentagi reference.

| # | Trigger | What it catches | Source |
|---|---|---|---|
| 1 | **Plan quality** | No recon before destructive; 3+ calls to same tool family in one turn; no `--target` set when a `targetRequired` tool is called | New in v0.2 |
| 2 | **Stuck / loop** | Same `(toolName, argsHash)` ≥4 times within the last 8 tool calls | v0.1 "interveneIfStuck" |
| 3 | (out of scope) | Overreach — tool call outside session target / scope | Chokepoint (v0.1) |
| 4 | **Confidence calibration** | Agent says "scan complete" with 0 findings; claim-without-evidence; closed-port claim for a port not in the scan range | v0.1 §9 "tool false-confidence" |
| 5 | (out of scope) | Risk escalation — destructive action with insufficient justification | Chokepoint (v0.1) |
| 6 | **Postmortem** | End-of-turn narrative: what we tried, what we learned, what's missing, next step | Pentagi monitor pattern |

**Excluded (explicitly):**
- **Trigger 3 (overreach)**: chokepoint's `checkTarget` rule already denies out-of-scope `targetRequired` calls (v0.1 polish). Duplicating this in the supervisor is redundant.
- **Trigger 5 (risk escalation)**: chokepoint's `checkDestructive` + `checkElevation` already require confirm/typed-confirm. A "soft escalation" supervisor is duplicate work.
- **LLM opt-in for trigger 1**: pure rule is high-confidence for "no recon before destructive" and "3+ same family". An LLM judge is a v0.3 stretch.

## 2. Behavior contract

The supervisor is **strictly silent** — it cannot interrupt the user
or block a tool call. Its three permitted actions:

1. **Advise**: inject a `role: 'user'` message into the agent's `history` array with a "Supervisor: ..." prefix. The next `streamText` step sees it. (For rules A/B/C only — the postmortem does not inject back into history.)
2. **Log**: write a `SupervisorFireRecord` to the session log's `supervisor.fires[]` array. Survives `--resume`.
3. **Emit**: yield a new `AgentEvent` variant to the consumer:
   - `supervisor-fire` (mid-turn, per rule fire, with a `targetEventId` referencing the offending event)
   - `supervisor-postmortem` (end-of-turn, one per turn)

The supervisor **cannot**:
- Throw, deny, or replace a `tool-call-request`
- Override the chokepoint's `deny` decision
- Block on a user prompt
- Mutate the agent's primary LLM call
- Re-write prompts, swap models, or otherwise change what the agent is asked to do this step
- Add latency to the agent's hot path (rules are synchronous; postmortem is async but does not gate the next turn)

**When the supervisor runs:**
- Rules A, B, C: synchronous, on every `AgentEvent`, per-step cost < 1ms
- Postmortem: async, fired on the `done` event, runs in parallel with the next turn's start (if the user is fast) or completes before the user types their next message (if they're not)

**Failure modes:**
- A rule firing incorrectly (false positive): logged as a `fire` record; the user sees a ⚠ marker on the offending event; the agent has already been advised (the advice is benign: "try a different tool" or "the scan may be incomplete")
- The postmortem LLM call failing: emits `supervisor-postmortem` with a fallback body `"Supervisor postmortem unavailable: {error}"`; never throws
- The supervisor itself crashing: `withSupervisor` catches and yields a single `error` event; the inner loop is unaffected (the wrapper re-yields everything unchanged)

## 3. Architecture

A new `packages/core/src/agent/supervisor.ts` exports a single function
`withSupervisor(opts)` that wraps `runTurn(opts)`. The wrapper:

```
withSupervisor({ runTurn, historyRef, session, model, … })
  → AsyncIterable<AgentEvent>
```

Internally:

1. Create a `SupervisorState` (counters, ring buffer, fire history).
2. For-await the inner `runTurn(...)` async iterable.
3. For each yielded event:
   a. Call `ruleEngine.observe(state, event)` (synchronous, no I/O)
   b. If the rule fired, call `applyFire({state, event, historyRef, session, fire})`:
      - Append a `SupervisorFireRecord` to `state.firesThisTurn`
      - Inject an advice `user` message into `historyRef` (mutated in place — the v0.1 `AgentApp` already supports this pattern via `setMessages(prev => [...prev, ...])`)
      - Yield a `supervisor-fire` `AgentEvent` to the consumer
   c. Re-yield the original event unchanged
4. On the `done` event, kick off `generatePostmortem({state, history, model})` in parallel (do not await before yielding `done` — postmortem arrives as a separate `supervisor-postmortem` event after `done`).
5. On end of stream, ensure the postmortem is awaited and yielded, then close.

**Why a wrapper, not a hook in `loop.ts`:** the v0.1 master plan §7
("single file with clear seams — v0.2 plugs in") describes a hook
shape, but the *actual* seam v0.1 ships is the `AsyncIterable<AgentEvent>`
stream — every event is a yielded value. The wrapper is **outside** the
loop, requires **zero changes to `loop.ts`**, and can be A/B tested by
wrapping or not wrapping. The hook shape would require new public
contracts on `runTurn` (`observe` + `advise`) for a benefit (real-time
vs. post-hoc) that doesn't change user-visible behavior.

## 4. Rule engine (triggers 1, 2, 4)

All three rules are pure functions over `(state, event) → state' | { state', fire? }`.
Lives in `packages/core/src/agent/supervisor-rules.ts`.

### 4.1 Rule A — Stuck / loop (trigger 2)

**State:** ring buffer of the last N=8 `(toolName, argsHash)` pairs.
```ts
type RuleAState = { recent: Array<{ name: string; argsHash: string; ts: number }> };
```

**Fire condition:** same `(toolName, argsHash)` appears **≥4 times within the last 8 `tool-call-request` events**.

**Why 4-of-8, not 3-of-5:** a real pentest often calls `whois` 2-3 times
in a session for different domains. 4-of-8 is high-confidence stuck;
3-of-5 would fire on legitimate "scanning multiple targets in parallel"
workflows. 4-of-8 has a clean falsifiable test
("5 `nmap -p 80 host` calls + 1 unrelated `whois` call → fires on the 5th").

**Output:** `{ kind: 'loop-detected', tool: string, count: number, recent: string[] }`.

**Advice text template:**
> "Supervisor: you called `{tool}` {count} times in the last {N} tool calls with the same arguments. Try a different approach — e.g. `{alt}`."

`alt` is from a small static table:

| Tool family | Alt suggestion |
|---|---|
| `nmap_*` | "scan fewer ports; add `-sV` for service detection; or move to a different host" |
| `whois` / `dig` | "you've already got the DNS records; try a different recon tool or move on" |
| `nuclei_*` / `nikto_*` | "the scan returned no findings; try a different template or a different tool" |
| `http_get` | "the page returned 200; check the response body for new endpoints or move on" |
| anything else | "try a different tool or a different target" |

### 4.2 Rule B — Confidence calibration (trigger 4)

**State:** sliding window of the last K=20 `text-delta` chunks + a "tool calls since last claim" counter.
```ts
type RuleBState = { recentText: string; toolCallsSinceLastClaim: number };
```

**Fire conditions (any one):**

1. **Empty-findings claim** — a `text-delta` matches `/(scan|recon|port[- ]?scan|enum(eration)?)\s+(is\s+)?(complete|done|finished)/i` AND the session's `findings.jsonl` has 0 entries for the host the scan targeted. Fire: "you said the scan is complete, but no findings were written to disk for `{host}`."
2. **Claim without evidence** — words `complete` / `done` / `finished` / `no vulnerabilities` / `nothing found` appear within 2 tool calls of the claim, AND the most-recent tool call returned an empty result (`{ok: true, output: []}` or `{ok: true, output: ''}`). Fire: "you said X, but the last tool call returned an empty result — was that a real conclusion or a guess?"
3. **Negative-result overconfidence** — a `text-delta` claims `port N is closed` but the most-recent `nmap_*` tool result shows N was not in the scanned port range. Fire: "you said port N is closed, but it wasn't in the scan range — port state is unknown, not closed."

**False-positive controls:**
- A `complete` claim within 1 turn of a `text-delta` that names a specific finding (`found CVE-2024-XXXX on /admin`) is *not* a fire.
- A `complete` claim that names a finding count (`3 findings: ...`) is *not* a fire.
- A `complete` claim in response to a user message (vs. a tool result) is *not* a fire.

**Output:** `{ kind: 'overclaim', quote: string, evidence: string }`.

### 4.3 Rule C — Plan quality (trigger 1)

**State:** small set of counters per turn.
```ts
type RuleCState = {
  toolsCalledThisTurn: number;
  destructiveCallsThisTurn: number;
  reconCallsThisTurn: number;
  distinctToolFamiliesThisTurn: Set<string>;
  chokepointSessionTarget?: string;
};
```

**Fire conditions:**

1. **No recon, going destructive** — `destructiveCallsThisTurn > 0 && reconCallsThisTurn === 0 && toolsCalledThisTurn > 1`. Fire: "you're about to run a destructive tool without any prior recon. Consider `nmap_scan` or `whois` first."
2. **Repeated same tool family** — 3+ calls to the same tool family (e.g. 3 different `nmap_*` calls) in a single turn. Fire: "3 different `nmap_*` calls in one turn. Consider a single comprehensive scan instead."
3. **No `--target` set + `targetRequired` tool called** — `chokepointSessionTarget` is undefined and a `targetRequired` tool was called. Fire: "this tool requires a target scope; consider `gmft --target <host>` to bind the session."

**Output:** `{ kind: 'plan-issue', severity: 'info' | 'warn', text: string }`.

### 4.4 Why three rules, not one mega-rule

- Each is independently testable with a fixture of canned events
- Each has a different update cadence (A is per-tool-call, B is per-text-delta, C is per-tool-call with a longer memory)
- Combining them would make the rule file ~500 lines and impossible to reason about

## 5. Postmortem generator (trigger 6)

Lives in `packages/core/src/agent/supervisor-postmortem.ts`. Single function:

```ts
async function generatePostmortem(input: {
  turn: TurnRecord;
  sessionFindings: Finding[];
  model: LanguageModel;
  system?: string;
}): Promise<string>
```

The postmortem is **one LLM call** with a fixed 4-section prompt:

```
You are a pentest postmortem writer. Given a single turn of agent activity
(scan calls, findings, the agent's final text reply), produce a 3-5 sentence
postmortem in this exact shape:

  1. WHAT WE TRIED: 1 sentence naming the tool calls made this turn.
  2. WHAT WE LEARNED: 1-2 sentences naming concrete findings (or "no
     new findings").
  3. WHAT WE'RE MISSING: 1-2 sentences naming the most obvious gap
     (e.g. UDP, service detection, vuln scan, brute force, …).
  4. NEXT STEP: 1 sentence suggesting the most useful next action.

Be specific. Reference tool names, port numbers, finding IDs. Do not
speculate beyond what the turn's evidence supports. Do not apologize.
Do not pad.

Turn evidence (in chronological order):
{events JSON}

Session findings so far:
{findings JSON}

Agent's final text reply:
{finalText}
```

**Output constraints:**
- 3-5 sentences, total ≤ 120 words
- Returns a `string` (no streaming)
- On LLM error: returns `"Supervisor postmortem unavailable: {error}"` — never throws
- Timeout: 10s, same fallback string

**Why a fixed prompt:** the postmortem is a low-stakes, structured
output. The 4-part shape is a contract the TUI can render predictably.
The prompt is small enough that any of the 5 v0.1 providers can produce
it well.

## 6. Advice injection (rule fires → next step's history)

Three constraints:

1. The supervisor's `for-await` consumer wraps the inner `runTurn` stream — it sees events as they're yielded, but the inner loop is already in flight.
2. The advice needs to land in the agent's `history` array *before* the next step's `streamText` — that's the only way the LLM sees it.
3. `history` is an array the agent loop is reading from — we can't safely mutate it mid-flight.

**Solution:** the `withSupervisor` wrapper takes the `history` array as a
parameter (a `ref`), holds a reference to it, and mutates it directly.
The `runTurn` loop reads `history` lazily — each `streamText` call
snapshots the current array. This is the v0.1 pattern: the existing
`AgentApp` does `setMessages(prev => [...prev, ...])` and the loop reads
the latest.

**Shape:** a `user` message with the literal prefix "Supervisor: ":
```ts
type SupervisorAdvice = {
  role: 'user';
  content: `Supervisor: ${string}`;
};
```

**Why `role: 'user'`, not `role: 'system'`:**
- v0.1's `runTurn` doesn't always support system messages in the history (the AI SDK's `streamText` interprets the role)
- v0.1's system prompt is set via `buildSystemPrompt('agent', env)` at boot, not per-turn
- A `user` message with the literal prefix "Supervisor: ..." is unambiguous to the agent, the TUI, and the audit log

Rules only fire *after* the offending event, so the advice is always
in a subsequent step's context. The exact step it lands in depends on
where in the turn the rule fires — which is the desired behavior (the
agent sees "supervisor says: try service detection" on the step
*after* the 5th identical `nmap` call).

## 7. `AgentEvent` extension

v0.1 ships 9 variants. v0.2.A adds 2:

```ts
// Mid-turn — one per rule fire
| {
    type: 'supervisor-fire';
    fire: SupervisorFire;       // discriminated union: {kind: 'loop-detected' | 'overclaim' | 'plan-issue', ...}
    targetEventId: string;      // the id of the tool-result / text-delta it annotates
  }

// End-of-turn — one per turn
| {
    type: 'supervisor-postmortem';
    body: string;
    turnId: string;
    fireCount: number;
  }
```

**Why two new variants, not `text-delta` reuse:**
- `text-delta` is the agent's *own* text — interleaving supervisor output into it would let the LLM mistake supervisor advice for its own previous output
- The TUI needs to render the postmortem differently (collapsible card, not a chat line)
- The audit log wants to know "this is a supervisor observation" distinctly from "this is the agent talking"
- Backward compatible (additive)

The `done` event gains an optional denormalized `postmortem?: string`
field for convenience (a TUI that doesn't want to handle the new
variant can read the `done` payload), but the canonical carrier is the
new variant.

## 8. TUI surface

Three small changes in `apps/gmft/src/`:

1. **`StatusRail`** gains a `Supervisor` field:
   - **Quiet turn** (0 fires): `supervisor: —`
   - **Fires this turn**: `supervisor: ⚠ {N} fires` (clickable → opens a side-rail with the fire log)
   - **Postmortem present**: `supervisor: ◷ postmortem ready` (clickable → expands the postmortem card)

2. **Inline ⚠ markers** on `tool-result` and `text-delta` lines that have an associated `supervisor-fire` event. The TUI's transcript render correlates the `targetEventId` and adds a yellow `⚠` prefix. Hover/keyboard reveals the full fire text.

3. **`SupervisorPostmortemCard`** — a collapsible block at the *end* of the turn's transcript. Default: collapsed, one-line summary (`Supervisor postmortem: 3 fires, 1 finding, missing UDP`). Expanded: the 3-5 sentence body. Collapsing is local state.

**No new tabs.** The supervisor lives in the existing Chat tab and (lightly) in the StatusRail. The Findings tab is unaffected.

## 9. Session-log schema migration

v0.1's session log (`~/.local/share/gmft/sessions/<id>/sessions.jsonl`):
```ts
type TurnRecordV1 = {
  id: string;
  ts: number;
  messages: ChatMessage[];
  // ...
};
```

v0.2.A adds:
```ts
type TurnRecordV2 = TurnRecordV1 & {
  supervisor?: {
    fires: SupervisorFireRecord[];   // the rule fires that occurred this turn
    postmortem?: string;             // the LLM-generated postmortem (if any)
    postmortemError?: string;        // set if the LLM call failed
    modelUsed?: string;              // which model produced the postmortem
  };
};
```

`SupervisorFireRecord` is the serialized form of the `SupervisorFire` discriminated union — same shape, all primitives or arrays of primitives, JSON-safe.

**Migration:**
- A new top-level field on the session record: `schemaVersion: 1 | 2`. v0.1 logs default to `1`; v0.2 writes `2`.
- `loadLog` reads `schemaVersion` and:
  - `1` → treat `supervisor` as `undefined` for every turn
  - `2` → parse `supervisor` normally
- `appendTurn` writes `2` and the `supervisor` field if any fires occurred
- A new `_migrationNote` field on the session record (one-time, on first v0.2 write to a v0.1 log) is **not** required — `loadLog`'s version check is sufficient

**Redaction (extends v0.1 1.5g):**
- `supervisor.fires[].quote` and `supervisor.postmortem` go through `redactSecrets` before being written
- A user pasting `sk-ant-1234567890abcdef` into chat, then the supervisor quoting it in an overclaim detection, results in `[REDACTED]` on disk in both the message log and the `supervisor.quote` field

## 14. Risks & mitigations

| Risk | Mitigation |
|---|---|
| Rule false positives annoy the user | False-positive controls in §4.2 (3) and §4.3 (3) reduce noise. Inline ⚠ markers are subtle (a single character); the agent's response is unaffected unless the rule actually fires. The postmortem is opt-out-able via a future `--supervisor-postmortem=false` flag (not in v0.2.A). |
| Postmortem LLM call adds latency to "user types next message" | The postmortem runs in parallel with the next turn's start. If the user is fast, the next turn's first `text-delta` waits for the postmortem (via `Promise.all` on a per-turn barrier). 10s timeout caps the wait. |
| Advice injection mutates `history` mid-loop | v0.1 already supports this pattern (`AgentApp`'s `setMessages` is called from event handlers). The supervisor holds the same array reference and pushes. Tests cover the "advice arrives before next step" sequencing. |
| Schema migration breaks `--resume` of v0.1 sessions | `loadLog` checks `schemaVersion` and treats v0.1 logs as `supervisor: undefined`. `appendTurn` writes `2` with the new field. Round-trip test (v0.1 log → v0.2 read → v0.2 write) confirms the migration is lossless for the original turn data. |
| New `AgentEvent` variants break TUI consumers that switch on the union | The variants are additive. The existing TUI's `switch(event.type)` falls through to a default render for unknown types. 4 new tests assert the unknown-type default. |
| Rule engine file grows to 500+ lines over time | The 3 rules are physically separated (`supervisor-rules.ts` with sub-functions). Each has its own test file. Adding a 4th rule is additive. |

## 15. Acceptance criteria

A phase is **done** when:

1. All its tasks are merged to `main`.
2. `pnpm -r test` is green.
3. `pnpm -r build` produces a runnable `apps/gmft/dist/cli.js`.
4. The CHANGELOG entry is written.
5. A `v0.2.0-A.N` git tag is pushed.

The `v0.2.0-A` release requires **all 3 phases** plus:
- `pnpm -r test` is green at 420+ tests.
- `pnpm -r build` is clean.
- `pnpm -r typecheck` is clean.
- A demo cast (asciinema) of `--target scanme.nmap.org` showing a loop detection and postmortem lands in `docs/screenshots/`.
- The existing v0.1 docs (safety.md, tool-catalog.md) are updated to mention the supervisor in the threat model.
