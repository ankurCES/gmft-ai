# v0.3 — Run polish + tool surface completion + audit observability

**Status:** Proposed
**Date:** 2026-06-17
**Author:** Ankur
**Related ADRs:** [0010](./0010-v0.2-a.3-implementation-decisions.md) §4 (AgentApp marker wiring — follow-up), [0011](./0011-host-sandbox-enforcement.md) (host-sandbox policy), [0009](./0009-multi-agent-supervisor.md) §47 §71 (v0.3 stretch items: LLM-judge for loop-detector + `--supervisor-model` opt-out), [0005](./0005-onboarding-and-model-selector.md) §183 (onboarding follow-ups)
**Related plan:** `2026-06-17-gmft-v0.3-run-polish-and-tool-surface.md`

## Context

v0.2 shipped the multi-agent supervisor (v0.2.A) and the
host-sandbox enforcement (v0.2.D). The repo is at 497 tests
green, 14 tools, 1 user-facing surface (the TUI), and 1
session-log format (`schemaVersion: 2`).

v0.2 close-out surfaced **two pre-existing bugs** that we
fixed (ESM `require` removal in 3 files; `stream.test.ts`
CI flake). It also surfaced **one known issue** that we
deliberately deferred: the compiled `dist/cli.js` artifact
can't be run with plain `node` (ESM extensionless imports
fail at runtime). The `dev` script (`tsx`) is the
documented run path.

The original v0.1 plan's v0.3 scope was "60+ tools, closing
the hexstrike-ai gap." That's unrealistic for a single slice
— 14 tools took 9 phases, so 60+ would take ~40 phases at
the same pace. We're better off **expanding the catalog
incrementally** in v0.3.B (10 more recon tools) and
**investing in the surrounding infrastructure** in v0.3.A
(run polish) and v0.3.C (audit observability).

## Goals

1. **v0.3.A — Run polish.** Fix the `dist/` runtime issue
   (real, not aspirational). Wire the supervisor-fire
   marker into the AgentApp message stream (ADR-0010's
   only remaining follow-up). Add the `--supervisor-model`
   opt-out for the postmortem (ADR-0009 stretch). Make
   the TUI surface the audit log so users can review what
   the agent did.
2. **v0.3.B — Recon expansion.** Add 10 more read-only
   recon tools (masscan, rustscan, subfinder, httpx,
   wpscan, dnsrecon, fierce, enum4linux, ldapsearch,
   snmp-check) and 3 wifi/wireless tools (bettercap,
   aircrack-ng full workflow, kismet). All read-only;
   the destructive surface is unchanged. The catalog
   grows 14 → 27 tools, hitting the v0.2 plan's
   "22 in v0.2" target.
3. **v0.3.C — Audit observability.** Add a hash-chained
   `audit.jsonl` (separate from the session log) with
   `gmft audit verify`, `gmft audit log`, and `gmft audit
   tail` subcommands. The hash chain gives operators
   tamper-evidence for compliance review.

## Non-goals

- **No new LLM providers** (Claude/GPT-4o remain the
  supported set; Ollama/local is v0.4).
- **No cloud-native deployment** (k8s, multi-tenant) — that
  was the original v0.3 stretch and is now v1.0 scope.
- **No binary exploitation tools** (gdb, pwntools, ghidra)
  — these were v0.2 in the original plan but are out of
  scope for the v0.3 increment.
- **No MSP/team mode** — v1.0.
- **No Tauri desktop wrapper** — explicitly out of scope.
- **No LLM judge for trigger 1 (loop-detected)** — ADR-0009
  stretch, deferred to v0.4 (needs a labeled eval set).

## Design

### v0.3.A — Run polish

#### A.1 — `dist/` runtime fix

The root cause: `tsconfig.base.json` uses
`moduleResolution: "Bundler"`, which lets TS resolve
extensionless imports. Node's ESM resolver doesn't. The
fix is one of:

- **Option A: switch to `moduleResolution: "NodeNext"`** +
  add `.js` extensions to all 52 relative imports. Pure
  TS change, no runtime cost. **Cons:** 52-line PR; touch
  every `import` + `export *`.
- **Option B: add an esbuild build step** that bundles
  `dist/cli.js` into a self-contained artifact with
  extensions resolved. **Cons:** adds a build dep, breaks
  the "no bundler" promise of the v0.1 plan.
- **Option C: ship `dist/cli.js` as a tsx wrapper** that
  shells to `tsx ./src/cli.tsx`. **Cons:** ships source
  with the artifact (mitigated by `.npmignore` or build
  pipeline).

**Decision: Option A.** The 52-import PR is mechanical,
keeps the build pure-TS, and the resulting `dist/` artifact
is a real runnable ESM module. The tests will catch
anything we miss (vitest resolves through TS so the
extensionless import will still work in tests — we'll add
a `node ./dist/cli.js --version` smoke test to CI to
guard the runtime).

#### A.2 — `SupervisorFireMarker` AgentApp wiring

ADR-0010 §4 documented the marker as the smallest
renderable unit; the AgentApp-level wiring (deciding
which messages get a marker) was deferred. v0.3.A ships
the wiring:

- AgentApp maintains a `Set<eventId>` of "supervised"
  event IDs (the event ID of the assistant turn that
  triggered the fire). When the message stream renders
  an assistant message whose event ID is in the set,
  it injects a `SupervisorFireMarker` between the message
  and the next one.
- The set is updated by the wrapped `lastFires()` call
  (already used for the rail's `fires` state).
- 1-2 tests for the wiring (the marker renders on the
  right message, the marker doesn't render when there
  are no fires).

#### A.3 — `--supervisor-model` opt-out

ADR-0009 §71: the postmortem uses the same model as the
agent's primary model. Some users want to use a cheaper
or local model for the postmortem (it's a fixed-prompt
4-section task; a Haiku or Ollama model is fine). v0.3.A
adds:

- `apps/gmft/src/cli.tsx` — new `--supervisor-model <id>`
  flag. When set, the postmortem is generated with this
  model instead of the primary.
- `packages/core/src/llm/model-factory.ts` — already
  supports a model ID, so no new API surface.
- 1 test for the wiring.

#### A.4 — TUI audit log viewer

Add a `FindingsTab`-style tab to the TUI for the audit
log. Read-only viewer with pagination, filter by
event/timestamp, color-coded by event type. This makes
the audit log observable to the user mid-session
(without it, you have to `cat ~/.local/share/gmft/
sessions/<id>.jsonl` after the session ends).

- 4-6 tests (render, paginate, filter, empty state).

### v0.3.C — Audit observability

The current session log (`~/.local/share/gmft/sessions/
<id>.jsonl`) is a flat JSONL of `Turn` records. It has
**no integrity guarantees** — a malicious or buggy tool
could append a line, and an operator reviewing the log
later can't tell if it's been tampered with. v0.3.C adds:

#### C.1 — `audit.jsonl` (hash-chained)

A separate log file: `~/.local/share/gmft/audit/audit.jsonl`.
Each line is an `AuditEvent`:

```typescript
type AuditEvent =
  | { kind: 'tool-call', tool: string, args: object, decision: 'allow' | 'deny' | 'confirm', reason: string }
  | { kind: 'tool-result', tool: string, ok: boolean, mode: RunnerMode, durationMs: number }
  | { kind: 'chokepoint-decision', tool: string, rule: string, decision: 'allow' | 'deny' | 'confirm' }
  | { kind: 'session-start', sessionId: string, target: string | null, modelId: string }
  | { kind: 'session-end', sessionId: string, turnCount: number, durationMs: number }
  | { kind: 'runner-mode', mode: RunnerMode, capabilities: RunnerCapabilities }
  | { kind: 'onboard', config: { model: string, theme: string } }
```

Each line carries a `prevHash: string` (the SHA-256 of
the previous line's canonical form) and a `hash: string`
(the SHA-256 of this line's canonical form). The chain
makes tampering detectable — a single changed line breaks
the chain from that point forward.

#### C.2 — `gmft audit verify`

Subcommand: `gmft audit verify`. Walks the audit log
line-by-line, recomputes the hash, compares to the
recorded hash, and reports:

```
✓ audit chain intact (1247 events, 0 broken)
  last event: 2026-06-17T19:23:45.123Z (tool-result, rustscan_scan)
```

If broken:

```
✗ audit chain BROKEN at line 847
  line 847 hash mismatch
  recorded: a83f3bb3c8...
  computed: f0e6c6e9a1...
  events 848..1247 cannot be verified
```

#### C.3 — `gmft audit log` and `gmft audit tail`

- `gmft audit log` — paginated viewer (default 50 events).
  `--since <iso>`, `--until <iso>`, `--kind <k>` filters.
- `gmft audit tail` — like `tail -f`, streams new events
  to stdout.

#### C.4 — Hash chain key

The hash chain is keyed on a machine-specific secret
stored in `~/.config/gmft/audit.key` (generated on first
audit event, mode 0600). The key is used as the HMAC
salt, not just SHA-256. This means a chain from one
machine is **not verifiable** on another machine without
the key — which is the correct security property
(operators can copy the log but not forge entries from
a different machine). The key file is itself backed up
to the existing secret store.

#### C.5 — Tests

- 8-10 tests: hash chain integrity (intact / broken /
  missing key / replaced line / appended line /
  truncated), `audit verify` CLI output shape, `audit
  log` pagination, `audit log` filters, `audit tail`
  streaming, HMAC key generation, key file mode 0600.

v0.3.A → v0.3.B → v0.3.C, each tagged and released
independently.

## Open questions

1. **Should the hash chain include the *content* of
   tool args/results, or just metadata?** Including the
   content is more tamper-evident but the log gets very
   large (a `nmap` scan can produce 10MB of stdout).
   Default: include only metadata (tool name, args shape,
   result ok/error, mode, duration, decision). The full
   content goes in the existing session log; the audit
   log is the *tamper-evident index*.
2. **Should the HMAC key be backed up to the secret
   store automatically?** The secret store is the
   existing pattern (API keys, tokens). Auto-backup
   means: if the user re-installs gmft, the audit log is
   still verifiable. Default: yes, auto-backup, with
   `--no-backup` opt-out.
3. **What's the relationship between the audit log and
   the session log?** They're parallel: the session log
   has the *full conversation*; the audit log has the
   *tamper-evident index* of significant events. The
   session log is what the TUI renders; the audit log is
   what operators review post-session.
4. **Should the audit log be opt-in?** No — compliance
   review is the default use case, and an opt-out
   `GMFT_DISABLE_AUDIT_LOG=true` is sufficient for
   privacy-conscious users.
