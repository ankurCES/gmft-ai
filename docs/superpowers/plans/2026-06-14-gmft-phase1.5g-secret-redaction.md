**Goal:** Stop secrets from ever reaching the session log on disk. Today `redactSecrets(line)` exists as a string-level helper but **is not wired into `appendTurn`** â€” a user pasting `sk-ant-12345` into chat is written verbatim to the JSONL file. v0.1 plan Â§10 line 572 calls this out as 1.5g: "scrubs `(api[_-]?key|token|secret)\s*=\s*\S+` patterns. Apply at write time."

**Why write-time, not read-time:** disk dumps, `cat`, `grep` on the log, OS-level indexing, accidental `git add` â€” all of those leak if redaction only runs on read. Write-time redaction is the stronger invariant. The UX cost (the user can't see the unredacted value in `/session load` either) is fine because they pasted it into chat *intentionally* â€” if they want to keep it, it belongs in `SecretStore`, not chat.

**What changes in this plan:**
1. Extend `redactSecrets(line)` with the JSON shapes that show up in a serialized `Turn.content` field (`"apiKey": "..."`, `"token": "..."`, etc.). The existing regexes cover env-style and header-style; we add the JSON variants.
2. Wire `appendTurn(path, turn)` to call `redactSecrets` on the serialized line before `appendFile`. New `appendTurnRaw(path, turn)` escapes redaction for tests / explicit internal use.
3. Add a regression test asserting the API key never appears in the log when the user pastes it in chat â€” both env-style (`apiKey=...`), JSON-style (`"apiKey": "..."`), and bare provider-prefix (`sk-ant-...`) shapes.

**What does NOT change:** `redactSecrets` is still a pure string-level function. No JSON parsing, no AST. The serialized turn line is a string, and the secret patterns are recognizable substrings; parsing the JSON would add complexity for no gain (and would break on malformed lines, which we already accept in `readLog`).

**Tech Stack:** TypeScript ESM, vitest 2.1. No new deps.

## Design decisions called out (push back if you disagree)

1. **Write-time, not read-time.** Plan Â§10 says "apply at write time" and that's stronger. Cost: the user can't see their own pasted key when they `/session load` later. That's fine â€” they typed it into chat, not into a vault; if they want to keep it, `SecretStore` is the right place.

2. **No JSON parsing inside `redactSecrets`.** The serialized turn line is `{"role":"user","content":"<escaped>"}\n`. We could parse and walk the AST, but that's heavier and would force `redactSecrets` to know the JSON shape (a coupling we don't want). The string-level approach catches all four target patterns (env, header, bare, JSON) with simple regexes.

3. **`appendTurnRaw` as a sibling, not a flag.** A `redact?: boolean` option on `appendTurn` would be one fewer function. But it makes the default path ambiguous to readers â€” every caller has to think "did I pass the flag?" `appendTurn` (always redacts) + `appendTurnRaw` (never redacts) is unambiguous. The "raw" name signals "you better know what you're doing."

4. **JSON-shape regex covers `"apiKey"`, `"api_key"`, `"token"`, `"secret"`.** The v0.1 plan only lists `api[_-]?key|token|secret` for the env-style pattern. We extend that to JSON because the same user concern (pasting a structured blob) applies. We do NOT cover every possible JSON secret name (`password`, `private_key`, `client_secret`) â€” false-positive risk is real and the explicit set is enough for v0.1.

5. **`redactSecrets` stays pure.** We could memoize or compile the regexes once, but `replace` with literal regexes is fast enough at v0.1 volumes. Premature optimization.

## Test plan (4 new tests)

```ts
// All in packages/core/test/session-log.test.ts

it('appendTurn scrubs a pasted sk-ant- key from the user content', async () => {
  const p = join(dir, 'turns.jsonl');
  await appendTurn(p, { role: 'user', content: 'my key is sk-ant-1234567890abcdef' });
  const text = await readFile(p, 'utf8');
  expect(text).not.toMatch(/sk-ant-1234567890abcdef/);
  expect(text).toMatch(/\[REDACTED\]/);
});

it('appendTurn scrubs a pasted JSON-shaped apiKey from the user content', async () => {
  const p = join(dir, 'turns.jsonl');
  await appendTurn(p, { role: 'user', content: 'here: {"apiKey": "sk-ant-abcdef0123"}' });
  const text = await readFile(p, 'utf8');
  expect(text).not.toMatch(/sk-ant-abcdef0123/);
  expect(text).toMatch(/\[REDACTED\]/);
});

it('appendTurn scrubs a pasted Authorization header from the user content', async () => {
  const p = join(dir, 'turns.jsonl');
  await appendTurn(p, { role: 'user', content: 'curl -H "Authorization: Bearer sk-openai-9999"' });
  const text = await readFile(p, 'utf8');
  expect(text).not.toMatch(/sk-openai-9999/);
  expect(text).toMatch(/Authorization: \[REDACTED\]/);
});

it('appendTurn leaves normal chat content untouched', async () => {
  const p = join(dir, 'turns.jsonl');
  const content = 'hi how are you? â€” fine, just writing some tests';
  await appendTurn(p, { role: 'user', content });
  const turns = await readLog(p);
  expect(turns[0]?.content).toBe(content);
});
```

The first three are the v0.1 plan's "3 patterns redacted" (1.5g acceptance). The fourth is a regression-guard so the redact pass doesn't over-eat. The pre-existing `redactSecrets` unit test stays â€” it tests the helper directly, the new tests test the wiring through `appendTurn`.

## What does NOT ship

- A more sophisticated JSON-aware redactor (regex on the serialized line is enough)
- Redaction of the LLM's response content (the v0.1 plan is silent on this; LLM responses can echo user-pasted content but the input is the user's responsibility to not paste secrets. v0.2 might add output-side redaction.)
- Redaction of `Turn.meta` (the type doc says "Not redacted" â€” meta is structured machine data, not user chat)
- Migration of pre-1.5g log files (old sessions with raw secrets stay on disk; user is expected to delete them. Calling that out in CHANGELOG.)
