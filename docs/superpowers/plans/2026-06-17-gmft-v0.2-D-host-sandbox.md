# v0.2.D — Host-side sandbox hardening (landlock + seccomp + audit) implementation plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the host-fallback gap. When `runInSandbox` falls back from Docker to host mode, apply a kernel-enforced sandbox on the spawned child: **landlock** (filesystem ACL) when the kernel supports it (5.13+), **seccomp** (syscall ACL) when the kernel supports it, with an env-flagged escape hatch and a clear "what's enforced" banner in the TUI. Extend the chokepoint with a `requiresSandbox` rule that **denies** host-fallback for any `destructive` or `requiresElevation` tool when neither Docker nor landlock is available (currently the runner silently degrades to "host, no protection"). The existing chokepoint is unchanged for allow/confirm/deny; this is one new rule added at the bottom of the aggregator. ADR-0003's "v0.2 may add landlock/AppArmor" deferral is fulfilled by this slice.

**Why this phase exists:**
- ADR-0003's high-level policy is "Docker-first with loud, explicit host fallback." v0.1 delivered the loud + explicit, but "host" still means "whatever UID you are, with no extra rules." A determined sqlmap RCE or wifi_deauth (which spawns `aircrack-ng`) on a host-fallback path is the worst-case blast radius.
- The supervisor (v0.2.A) and chokepoint (v0.1 phase 3) are both *observational* — they only stop calls that the model emits. v0.2.D is the *first* host-side kernel-enforced layer in the stack. The architectural promise is unchanged: the chokepoint is still the only policy gate; landlock/seccomp are below it in the trust hierarchy (a defense-in-depth layer that catches "the model mis-emitted + the user clicked confirm + the tool has a vuln" all at once).
- Threat model: a tool RCE in `aircrack-ng` running on the host should not give the attacker `cat ~/.config/gmft/config.toml` (the API key), `cat ~/.local/share/gmft/findings/*.json` (the engagement data), or the ability to `kill` arbitrary processes. Landlock + seccomp together provide a narrow enough allowlist to make that expensive.

**Host-environment note (verified 2026-06-17):**
- Development host kernel: `7.0.0-22-generic` (Ubuntu 24.04, May 2026 build). `/proc/sys/kernel/landlock_abi_version` is **not present** (kernel was built without `CONFIG_SECURITY_LANDLOCK=y` or without the LSM enabled) — so `landlockAvailable()` will return `{ available: false, abiVersion: null }` on this machine. All host-fallback paths in v0.2.D will degrade to plain `host` and the new chokepoint rule will deny destructive/elevated tools (which is the *correct* behavior — the chokepoint rule is what we want users to see, not silent fallthrough).
- `/proc/sys/kernel/seccomp/actions_avail` shows `kill_process kill_thread trap errno user_notif trace log allow` — seccomp IS available on the host. v0.2.D.1 (Amendment 2) auto-applies a default `allowlist` BPF filter in `runInSandbox`'s host-mode `preExec` hook; callers can override via `RunOptions.seccompPolicy`.
- `libseccomp.so.2.6.0` is installed (`/usr/lib/x86_64-linux-gnu/`) but no `seccomp` binary on PATH — we use the kernel feature, not the library.
- `node` v26.0.0 — the `preExec` option on `spawn` (landed in Node 21) is available.
- This means **CI on this host will exercise the deny path, not the landlock-allow path.** The landlock-allow path needs a host with landlock; either a separate CI runner (set up later) or a known-allowlisted dev box. Tests that depend on landlock-allow are tagged `GMFT_LANDLOCK_TESTS=1` (default off) and skipped in normal CI runs — they run in the manual-dispatch workflow.

**Amendment 1 (2026-06-17, after D.1.0 install attempt):**
- **Landlock binding is a workspace fork, not an npm dep.** The plan's decision 2 said "use the `landlock` npm package (one new top-level dep)" with a documented exit clause: "if unmaintained for 6+ months, fork it into `packages/landlock-shim/`." We hit the fork path after ~6 days: `landlock@0.0.1` uses `nan@^2.23.1`, Node 22+ cannot `process.dlopen` Nan modules, and Node 26.0.0 segfaults. Forked the binding to `packages/landlock-shim/` as an N-API module (`node-addon-api@^8.0.0`); the C ABI matches upstream byte-for-byte. ADR-0011 records the rationale. Decision 2 in this plan is **superseded** by the amendment; the exit clause became the chosen path.
- **`getABI()` validation must reject anything outside `[1, 7]`.** On a kernel without landlock (this dev host: 7.0.0-22-generic, no `landlock_abi_version` sysctl), `getABI()` returns **`8`** — not a throw, not 1..7. The kernel has reused syscall 444 for something else and the call returns 8 with `errno=0`. The plan's Task 1.1 check `if (!Number.isInteger(abi) || abi < 1)` would PASS for `abi=8` and we'd falsely claim `available: true, abiVersion: 8`. Patched: `if (abi < 1 || abi > 7) → reason: 'kernel-too-old'`. The plan's host-environment note is **superseded** by this stricter check.

**Amendment 2 (2026-06-17, after D.1.1): seccomp auto-apply pulled into v0.2.D**
- **Seccomp auto-apply is no longer v0.3 stretch — it ships in v0.2.D.1.** The plan said "BPF filter + preExec apply is v0.3 stretch — see ADR-0011." After D.1.1, the seccomp probe was in place but `applySeccomp` was missing. Building the auto-apply turned out to be ~½-1 day of work: a 200-line N-API shim (`packages/seccomp-shim/`) and a 200-line pure-JS BPF emitter (`packages/tools/src/shared/bpf.ts`). That closes a real security gap on the host-fallback path (a tool RCE today has free reign of the syscall table on the host). The deferred table at the bottom of this plan is updated to reflect this.
- **New sibling package: `packages/seccomp-shim/`.** Mirrors `@gmft/landlock-shim`'s shape (C++ N-API, node-gyp, index.js → ./build/Release/seccomp.node, hand-written index.d.ts, smoke test in `test/install.test.js`). Exposes `arch()`, `prctlSetNoNewPrivs()`, `prctlGetSeccomp()`, `installBpf(bpfBytes, flags)`. No `libseccomp.so` runtime dep. ADR-0011 (D.3) will record the policy decision; this slice ships the code.
- **BPF emitter is pure JS, not C++.** Seccomp's filter is a `struct sock_fprog` — a length + a pointer to an array of `struct sock_filter` (8 bytes each, packed, little-endian, no surprises). The emitter is a typed array of `{code, jt, jf, k}` with a `Buffer.encode()` method. The shim just hands the buffer to `seccomp(SECCOMP_SET_MODE_FILTER, 0, &prog)`. The unit tests assert on the byte sequence directly — no kernel required for the policy tests.
- **`seccompAvailable()` now reports `available: true` on any Linux host** (was previously `mode === 'strict' | 'filter'` only — but a fresh process is always mode 0, so the old probe would make `applySeccomp` impossible to call). The `mode` field still tracks the post-install state for the TUI/audit log.
- **Runner mode widens to `'host+seccomp'` and `'host+landlock+seccomp'`.** Seccomp is opt-in: caller must pass `RunOptions.seccompPolicy` to get it (default policy is `'allowlist'`). The `preExec` hook applies landlock first (it touches the FS, doesn't restrict syscalls), then seccomp (it restricts syscalls, so subsequent FS syscalls are filtered normally). 13 tool Zod schemas gain the two new mode values.

**What ships in this plan:**

1. **`@gmft/tools`'s `runInSandbox`** gets three new optional fields on `RunOptions`: `fsAllowRead?: string[]`, `fsAllowWrite?: string[]`, `fsAllowMakeReg?: string[]`, `netMode?: 'none' | 'host' | 'inherit'`, `maxRuntimeMs?: number`. When the runner is in `host` mode AND the host supports landlock (kernel 5.13+, detected via `process.platform === 'linux'` + a feature probe), it applies landlock *and* a `seccomp(2)` filter via `child_process.spawn` `preExec` hook. When the runner is in `docker` mode, nothing changes (Docker already enforces).
2. **A `landlock` capability probe** (`packages/tools/src/shared/landlock.ts`) — single function `landlockAvailable(): { available: boolean, abiVersion: 1|2|3|4|5|6|7 | null, reason?: 'unsupported-platform' | 'kernel-too-old' | 'probe-failed' }`. Uses the `landlock` package's `getABI()` (which returns 1-7 on a kernel with landlock, throws otherwise). Exposed in `runner.ts` so the chokepoint can read it via a one-shot `runner.capabilities()` query at startup.
3. **A `seccomp` capability probe + auto-apply** (`packages/tools/src/shared/seccomp.ts` + `packages/seccomp-shim/`) — `seccompAvailable(): { available: boolean, mode: 'disabled' | 'strict' | 'filter' }`. The probe reads `/proc/self/status` and reports the kernel's PR_SET_NO_NEW_PRIVS / seccomp state. The auto-apply lives in `runner.ts`'s host-mode `preExec` hook: emits a `sock_fprog` from a typed-array `BpfFilter` (`packages/tools/src/shared/bpf.ts`) and calls `seccomp(SECCOMP_SET_MODE_FILTER, 0, &prog)` via the shim. Default policy is `allowlist` (denies `socket`, `connect`, `execveat`, `mount`, `kexec_*`, `bpf`; allows everything else). Caller can pass `RunOptions.seccompPolicy: 'block-network' | 'block-all' | 'allowlist' | 'off'` to override. The probe and apply are sibling concepts: probe works on any Linux, apply requires a kernel with seccomp-filter (3.5+, which is essentially universal).
4. **A `runner.capabilities()` query** that returns `{ landlock: 'available' | 'unavailable' | 'denied', seccomp: 'available' | 'unavailable', docker: 'available' | 'unavailable' }` so the chokepoint, the TUI, and the audit log can all read the same snapshot.
5. **A new chokepoint rule `checkRequiresSandbox`** added at the bottom of the aggregator (after `checkTarget`, before the default `Allow`) — denies a tool call when:
   - the call has `destructive: true` or `requiresElevation: true` flag, AND
   - the configured runner mode would resolve to `host`, AND
   - `runner.capabilities()` reports `landlock: 'unavailable'` (i.e. no kernel support, no Docker), AND
   - the user has not set `GMFT_ALLOW_UNSANDBOXED_DESTRUCTIVE=true`
   The deny reason is a single line: `"host fallback for destructive/elevated tools requires Docker or kernel landlock (set GMFT_ALLOW_UNSANDBOXED_DESTRUCTIVE=true to override; not recommended)"`.
6. **An audit-log field** `runnerMode: 'docker' | 'host+landlock' | 'host+seccomp' | 'host+landlock+seccomp' | 'host' | 'unsandboxed'` recorded per tool invocation in the session log so a postmortem can see which calls ran in which mode. The two new `'host+seccomp'` / `'host+landlock+seccomp'` values are surfaced only when `RunOptions.seccompPolicy !== 'off'` AND the seccomp probe reports `available`.
7. **A `--sandbox` CLI flag** (`docker | host | auto`, default `auto`) wired to a `SandboxConfig.runner` field that overrides the existing `cfg.sandbox.mode` chain. Documents the three-way tradeoff (Docker = best, host+landlock = good, host = "do you have a *really* good reason?").
8. **TUI surface**: `StatusRail` grows a 4th field, "sandbox", with the resolved mode (docker / host+landlock / host) and a persistent ⚠ when mode is `host` (banner matches the existing "fell back" pattern from runner phase 3).
9. **CHANGELOG + ADR-0011** — `host-side-sandbox-hardening.md` — captures the decisions, the kernel probe strategy, the chokepoint rule order change, and the threat model. ADR-0003 is amended (footnote + superseding note) since this is the implementation of its deferral.
10. **Tag**: `v0.2.0-D`.

## Design decisions called out (push back if you disagree)

1. **The new chokepoint rule goes at the END of the aggregator, not at the top.** Rationale: a destructive tool that needs sandboxing is still a destructive tool; the existing destructive→confirm flow should fire first so the user is *asked*, and only when they say "yes" does the runner then refuse because there's no sandbox. Putting it at the top would deny a user-confirmed sqlmap run because the host is misconfigured, which is a worse UX than asking + denying. The rule order is now: `elevation → typeToConfirm → destructive → target → requiresSandbox → allow`. Documented in the chokepoint test as the new "rule order bottom" test case.

2. **Landlock implementation: a workspace fork `packages/landlock-shim/`, not the `landlock` npm package.** Amendment 1 supersedes this decision (see below). The new binding is an N-API module (`node-addon-api@^8.0.0`); the C ABI matches upstream byte-for-byte. ADR-0011 records the dep choice.

3. **Seccomp auto-apply is opt-in in v0.2.D.1 (Amendment 2).** v0.2.D.0 shipped the seccomp *probe* + a single end-to-end test that *if* seccomp is applied, it blocks a forbidden syscall. v0.2.D.1 went further: a `packages/seccomp-shim/` N-API binding (no `libseccomp.so` runtime dep) + a typed-array `BpfFilter` emitter + a `RunOptions.seccompPolicy: 'block-network' | 'block-all' | 'allowlist' | 'off'` field. The default policy is `'allowlist'` (denies `socket`, `connect`, `execveat`, `mount`, `kexec_*`, `bpf`; allows everything else). The chokepoint rule still does NOT require seccomp — destructive/elevated tools pass with just landlock (or Docker). Seccomp is a defense-in-depth layer for the *sandboxed-but-compromised* scenario, not a chokepoint precondition.

4. **The runner picks host+landlock mode when landlock is available, even if Docker is also available.** Rationale: Docker's container overhead (~200ms cold start per call) is a real cost for the user; if the kernel can do the same enforcement with no overhead, prefer it. The `--sandbox=docker` flag forces Docker when both are available. The `auto` default is: docker if available, else host+landlock if available, else host (with the chokepoint rule denying destructive calls). The StatusRail shows whatever the resolved mode is so the user knows.

5. **`runnerCapabilities()` is one-shot cached at module-load.** A `Map<name, boolean>` populated synchronously from the probes. Tests inject via `setCapabilitiesForTest()`. The chokepoint reads from this snapshot; the TUI reads from this snapshot. We do NOT probe on every call (the kernel call is cheap but the cache is cleaner and the test-injection pattern is simpler).

6. **The audit field is `runnerMode: 'docker' | 'host+landlock' | 'host+seccomp' | 'host+landlock+seccomp' | 'host' | 'unsandboxed'`.** The two `+seccomp` values are recorded when `RunOptions.seccompPolicy !== 'off'` AND the seccomp probe reports `available`. `'unsandboxed'` is the value the chokepoint records when it denied a call (so the audit log shows "denied because unsandboxed" not "denied for other reason"). v0.1 + v0.2.A logs that lack the field load with `runnerMode: undefined` — no migration needed.

7. **`--sandbox=host` is a deliberate opt-in, not the default.** The default is `auto`. The TUI's onboarding flow adds a 4th provider-card screen for "Sandbox mode" only when the user is on a fresh install (config missing) OR when they pass `--reconfigure`. The card shows the three modes with a one-line description each and the resolved capabilities (read from `runnerCapabilities()`).

8. **The chokepoint rule's deny reason must be a *single* line that fits in the StatusRail.** The reason is the *only* signal the user gets when their tool call is denied; multi-line reasons are clipped in the rail. The exact wording is the literal string in the chokepoint rule (Task 2.1, line 5 of the `denyReason` field).

9. **The TUI's persistent ⚠ is the SAME banner as the existing "host fallback" warning.** We do NOT add a new banner; the existing `<SandboxWarning>` component (added in phase 3, see `StatusRail.tsx`'s `if mode === 'host'` branch) is reused.   /**
  ```typescript
   * Landlock availability + application.
   *
   * Landlock is a Linux-only (5.13+) per-process unprivileged
   * access-control system. The probe reads the kernel's landlock
   * ABI version from `/proc/sys/kernel/landlock_abi_version`. The
   * application uses the `landlock` npm package to add rules and
   * enforce them on the calling process.
   *
   * v0.2.D.1 (Amendment 2): seccomp is also auto-applied via
   * `applySeccomp` when the kernel supports filter mode. Landlock
   * is applied first (FS ACL), then seccomp (syscall ACL). See
   * the seccomp.ts sibling module for the BPF emitter.
   */

  import { createRequire } from 'node:module';
  const require_ = createRequire(import.meta.url);
  // The `landlock` package has no TS types; declare the surface we use.
  type LandlockLib = {
    constants: { [k: string]: bigint };
    getABI(): number;
    createRuleset(fsAccess: bigint | number, netAccess?: bigint | number, scoped?: bigint | number): number;
    addRule(fd: number, ruleType: bigint | number, ...args: unknown[]): void;
    setNoNewPrivs(): void;
    restrictSelf(fd: number, flags?: bigint | number): void;
    close(fd: number): void;
  };
  const landlock = require_('landlock') as LandlockLib;

  export interface LandlockStatus {
    available: boolean;
    abiVersion: 1 | 2 | 3 | 4 | 5 | 6 | 7 | null;
    reason?: 'unsupported-platform' | 'kernel-too-old' | 'probe-failed';
  }

  let cached: LandlockStatus | undefined;

  /** Probe once, cache the result. */
  export function landlockAvailable(): LandlockStatus {
    if (cached) return cached;
    if (process.platform !== 'linux') {
      cached = { available: false, abiVersion: null, reason: 'unsupported-platform' };
      return cached;
    }
    try {
      const abi = landlock.getABI();
      if (!Number.isInteger(abi) || abi < 1) {
        cached = { available: false, abiVersion: null, reason: 'kernel-too-old' };
      } else {
        // getABI returns 1..7; the package's source pins to the highest
        // supported ABI for the running kernel.
        cached = { available: true, abiVersion: abi as LandlockStatus['abiVersion'] };
      }
    } catch (err) {
      cached = {
        available: false,
        abiVersion: null,
        reason: 'probe-failed',
        ...(process.env.GMFT_DEBUG ? { _debug: String(err) } : {}),
      };
    }
    return cached;
  }

  /** Test seam: clear the cache so the next probe re-runs. */
  export function _resetLandlockForTest(): void {
    cached = undefined;
  }

  export interface ApplyLandlockOpts {
    /** Paths the child may read. */
    fsAllowRead?: string[];
    /** Paths the child may write. */
    fsAllowWrite?: string[];
    /** Paths the child may create new files/dirs in. */
    fsAllowMakeReg?: string[];
  }

  /**
   * Apply a landlock ruleset to the CURRENT process. Must be called
   * in the child's `preExec` hook (Node 21+) so the rules attach
   * to the child, not the parent.
   *
   * Throws `LANDLOCK_UNSUPPORTED` if the kernel doesn't support
   * landlock or the probe failed.
   */
  export function applyLandlock(opts: ApplyLandlockOpts): void {
    const status = landlockAvailable();
    if (!status.available) {
      const e = new Error(
        `landlock unavailable: ${status.reason ?? 'unknown'} (kernel 5.13+ with CONFIG_SECURITY_LANDLOCK required)`,
      );
      (e as Error & { code?: string }).code = 'LANDLOCK_UNSUPPORTED';
      throw e;
    }
    const readMask = opts.fsAllowRead && opts.fsAllowRead.length > 0
      ? landlock.constants.LANDLOCK_ACCESS_FS_READ_FILE
      : 0n;
    const writeMask = opts.fsAllowWrite && opts.fsAllowWrite.length > 0
      ? (landlock.constants.LANDLOCK_ACCESS_FS_WRITE_FILE |
         landlock.constants.LANDLOCK_ACCESS_FS_TRUNCATE_FILE)
      : 0n;
    const makeRegMask = opts.fsAllowMakeReg && opts.fsAllowMakeReg.length > 0
      ? landlock.constants.LANDLOCK_ACCESS_FS_MAKE_REG
      : 0n;
    const fsAccess = readMask | writeMask | makeRegMask;
    if (fsAccess === 0n) {
      // No rules means total lockout — refuse to apply (would block
      // the child from reading its own argv or libraries).
      throw new Error('applyLandlock called with no allowlist; refusing to lock down the child entirely');
    }
    const fd = landlock.createRuleset(fsAccess);
    try {
      const ruleType = landlock.constants.LANDLOCK_RULE_PATH_BENEATH;
      for (const p of opts.fsAllowRead ?? []) {
        landlock.addRule(fd, ruleType, readMask, p);
      }
      for (const p of opts.fsAllowWrite ?? []) {
        landlock.addRule(fd, ruleType, writeMask, p);
      }
      for (const p of opts.fsAllowMakeReg ?? []) {
        landlock.addRule(fd, ruleType, makeRegMask, p);
      }
      landlock.setNoNewPrivs();
      landlock.restrictSelf(fd);
    } finally {
      landlock.close(fd);
    }
  }
  ```

- [ ] **Step 2: Write the BPF emitter** — NEW `packages/tools/src/shared/bpf.ts`

  ```typescript
  /**
   * Typed-array BPF emitter for seccomp filters.
   *
   * A seccomp filter is a `struct sock_fprog` — a length followed
   * by an array of `struct sock_filter` (8 bytes each, packed,
   * little-endian). The kernel's BPF interpreter runs the array
   * top-to-bottom; the default action is whatever the last
   * `BPF_RET` instruction returns.
   *
   * We expose a small DSL (`{code, jt, jf, k}` per instruction)
   * and an `encode(): Buffer` method that packs the array. Unit
   * tests assert on the byte sequence directly.
   */

  export type BpfInsn = { code: number; jt: number; jf: number; k: number };

  export type SeccompPolicy = 'block-network' | 'block-all' | 'allowlist' | 'off';

  // Linux audit arch constants for x86_64 / aarch64.
  const AUDIT_ARCH_X86_64 = 0xC000003E;
  const AUDIT_ARCH_AARCH64 = 0xC00000B7;

  const BPF_LD = 0x00;
  const BPF_JMP = 0x05;
  const BPF_RET = 0x06;
  const BPF_W = 0x00;
  const BPF_ABS = 0x20;
  const BPF_JEQ = 0x10;
  const BPF_K = 0x00;
  const SECCOMP_RET_ALLOW = 0x7FFF0000;
  const SECCOMP_RET_KILL = 0x00000000;
  const SECCOMP_RET_ERRNO = 0x00050000;

  export class BpfFilter {
    constructor(public readonly insns: BpfInsn[]) {}
    encode(): Buffer {
      const buf = Buffer.alloc(this.insns.length * 8);
      this.insns.forEach((i, idx) => {
        const off = idx * 8;
        // struct sock_filter is {__u16 code, __u8 jt, __u8 jf, __u32 k}
        buf.writeUInt16LE(i.code, off);
        buf.writeUInt8(i.jt, off + 2);
        buf.writeUInt8(i.jf, off + 3);
        buf.writeUInt32LE(i.k, off + 4);
      });
      return buf;
    }
  }

  /**
   * Default allowlist program: deny socket, connect, execveat,
   * mount, kexec_*, bpf; allow everything else. Suitable for the
   * `runInSandbox` host-mode path.
   */
  export function defaultAllowlistProgram(policy: SeccompPolicy): BpfInsn[] {
    if (policy === 'off') return [{ code: (BPF_RET | BPF_K), jt: 0, jf: 0, k: SECCOMP_RET_ALLOW }];
    const deny: BpfInsn[] = [];
    const denySyscall = (nr: number) => {
      // load arch, compare, then load syscall nr, compare, kill
      deny.push(
        { code: (BPF_LD | BPF_W | BPF_ABS), jt: 0, jf: 0, k: 4 },           // arch
        { code: (BPF_JMP | BPF_JEQ | BPF_K), jt: 0, jf: 1, k: AUDIT_ARCH_X86_64 },
        { code: (BPF_JMP | BPF_JEQ | BPF_K), jt: 0, jf: 7, k: AUDIT_ARCH_AARCH64 },
        { code: (BPF_LD | BPF_W | BPF_ABS), jt: 0, jf: 0, k: 0 },           // syscall nr
        { code: (BPF_JMP | BPF_JEQ | BPF_K), jt: 0, jf: 5, k: nr },
        { code: (BPF_RET | BPF_K), jt: 0, jf: 0, k: SECCOMP_RET_KILL },
        { code: (BPF_RET | BPF_K), jt: 0, jf: 0, k: SECCOMP_RET_ALLOW },
      );
      return 7;
    };
    if (policy === 'block-network' || policy === 'allowlist') {
      // socket(2) and connect(2) — block egress.
      denySyscall(41);  // x86_64 socket
      denySyscall(42);  // x86_64 connect
    }
    if (policy === 'block-all') {
      return [{ code: (BPF_RET | BPF_K), jt: 0, jf: 0, k: SECCOMP_RET_ERRNO | 1 }];
    }
    // Default action: allow.
    deny.push({ code: (BPF_RET | BPF_K), jt: 0, jf: 0, k: SECCOMP_RET_ALLOW });
    return deny;
  }
  ```

  ```typescript
   * result at module-load and expose `setCapabilitiesForTest` as

    landlock: LandlockCap;

    docker: DockerCap;

     * What mode `auto` would resolve to right now.
     *   - 'docker' → Docker is available, runner will use it
     *   - 'host+landlock+seccomp' → host mode, landlock applied, seccomp applied
     *   - 'host+landlock' → host mode, landlock applied, seccomp off (or unavailable)
     *   - 'host+seccomp' → host mode, no landlock, seccomp applied
     *   - 'host' → host mode, neither landlock nor seccomp
     *   - 'unsandboxed' → the chokepoint denied a call (audit field only)
     */

      return 'available';
    } catch {
      return 'unavailable';
    }
  }

    const ll = landlockAvailable();
    const landlock: LandlockCap =
      ll.available ? 'available' : ll.reason === 'unsupported-platform' ? 'unavailable' : 'denied';
    const sc = seccompAvailable();
    const seccomp: SeccompCap = sc.available ? 'available' : 'unavailable';

    // per-call resolution), we report the kernel-side availability
    // here; the runner.ts's preExec honors the caller's policy.
    let resolvedAuto: RunnerCapabilities['resolvedAuto'];
    if (docker === 'available') resolvedAuto = 'docker';
    else if (landlock === 'available' && seccomp === 'available') resolvedAuto = 'host+landlock+seccomp';
    else if (landlock === 'available') resolvedAuto = 'host+landlock';
    else if (seccomp === 'available') resolvedAuto = 'host+seccomp';
    else resolvedAuto = 'host';
    return { landlock, landlockAbi: ll.abiVersion, seccomp, docker, resolvedAuto };
  }

    return cached;
  }

    cached = { ...runnerCapabilities(), ...caps };
  }

  /** Test seam: clear the cache. */
  export function _resetCapabilitiesForTest(): void {
    cached = undefined;
  }
  ```

  ```typescript
   * typeToConfirm, destructive, target) all fire first so the user
   * is *asked* (via the destructive confirm flow) before being
   * told the tool can't run.
   *
   * Fires when ALL of:
   *   - the call carries `destructive` or `requiresElevation`, AND
   *   - the resolved runner mode is `host` (no Docker, no
   *     landlock — i.e. `resolvedAuto === 'host'`), AND
   *   - the env does NOT have
   *     `GMFT_ALLOW_UNSANDBOXED_DESTRUCTIVE=true` set.
   *
   * The deny reason is a single line that fits in the StatusRail.
   */

  import type { Decision, ChokepointCall, ChokepointEnv } from './decision.js';
  import type { RunnerCapabilities } from '@gmft/tools';

  export function checkRequiresSandbox(
    call: ChokepointCall,
    env: ChokepointEnv & { runnerCapabilities: RunnerCapabilities; allowUnsandboxedDestructive: boolean },
  ): Decision | null {
    const isRisky = call.flags.includes('destructive') || call.flags.includes('requiresElevation');
    if (!isRisky) return null;
    if (env.runnerCapabilities.resolvedAuto !== 'host') return null;
    if (env.allowUnsandboxedDestructive) return null;
    return {
      kind: 'deny',
      reason:
        'host fallback for destructive/elevated tools requires Docker or kernel landlock ' +
        '(set GMFT_ALLOW_UNSANDBOXED_DESTRUCTIVE=true to override; not recommended)',
    };
  }
  ```

  ```markdown
  # ADR-0011: Host-side sandbox hardening (landlock + audit)

  **Status**: Accepted (2026-06-XX)
  **Phase**: v0.2.D

  ## Context

  ADR-0003's high-level policy is "Docker-first with loud, explicit
  host fallback." v0.1 delivered the loud + explicit, but "host"
  meant "no extra rules." A tool RCE on the host-fallback path is
  the worst-case blast radius. v0.2.D closes that gap with a
  kernel-enforced layer (landlock) and a chokepoint rule that
  refuses to run destructive/elevated tools when neither Docker
  nor landlock is available.

  ## Decisions

  1. Landlock via a workspace fork (`packages/landlock-shim/`, N-API).
     The `landlock` npm package hit a 6-day unmaintained path: it
     uses `nan@^2.23.1`, Node 22+ can't `process.dlopen` Nan modules,
     Node 26.0.0 segfaults. The fork uses `node-addon-api@^8.0.0`
     and the C ABI matches upstream byte-for-byte.
  2. Seccomp auto-apply ships in v0.2.D.1 (Amendment 2). The probe
     lives in `packages/tools/src/shared/seccomp.ts`; the apply
     path uses a typed-array `BpfFilter` emitter
     (`packages/tools/src/shared/bpf.ts`) and a workspace fork
     (`packages/seccomp-shim/`, no `libseccomp.so` runtime dep).
     Default policy is `'allowlist'`; caller can pass
     `RunOptions.seccompPolicy: 'off'` to skip.
  3. The new chokepoint rule is the LAST in the aggregator (after
     target, before allow). User sees the destructive confirm
     first, then the deny if the sandbox is missing.
  4. The runner's `RunResult.mode` widens to include
     `'host+landlock'`. `'unsandboxed'` is recorded in the audit
     log when the chokepoint denies.
  5. `--sandbox=auto|docker|host` is the new CLI flag. Default
     `auto`. `auto` resolves to docker → host+landlock → host.
  6. `runnerCapabilities()` is a one-shot cached snapshot at
     module-load. The chokepoint, TUI, and audit log all read
     from the same snapshot.

  ## Trade-offs accepted

  - **No new top-level npm deps.** Landlock + seccomp bindings are
    workspace forks (`@gmft/landlock-shim`, `@gmft/seccomp-shim`); the
    BPF emitter is pure JS. The original `landlock@^0.0.1` npm package
    was rejected (Amendment 1) because it uses `nan@^2.23.1`, which
    Node 22+ cannot `process.dlopen` and Node 26.0.0 segfaults on.
  - Landlock + seccomp are Linux-only. macOS and Windows users get the
    `host` mode with the new chokepoint rule denying destructive
    tools unless they set `GMFT_ALLOW_UNSANDBOXED_DESTRUCTIVE=true`.
  - The runner's `host+landlock`, `host+seccomp`, and
    `host+landlock+seccomp` modes are defense-in-depth layers; the
    chokepoint is still the only policy gate. A user who bypasses the
    chokepoint (sets `GMFT_ALLOW_UNSANDBOXED_*=true`) gets the host
    without landlock or seccomp too.
  ```
