# v0.2.D — Host-side sandbox hardening (landlock + seccomp + audit) implementation plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the host-fallback gap. When `runInSandbox` falls back from Docker to host mode, apply a kernel-enforced sandbox on the spawned child: **landlock** (filesystem ACL) when the kernel supports it (5.13+), **seccomp** (syscall ACL) when the kernel supports it, with an env-flagged escape hatch and a clear "what's enforced" banner in the TUI. Extend the chokepoint with a `requiresSandbox` rule that **denies** host-fallback for any `destructive` or `requiresElevation` tool when neither Docker nor landlock is available (currently the runner silently degrades to "host, no protection"). The existing chokepoint is unchanged for allow/confirm/deny; this is one new rule added at the bottom of the aggregator. ADR-0003's "v0.2 may add landlock/AppArmor" deferral is fulfilled by this slice.

**Why this phase exists:**
- ADR-0003's high-level policy is "Docker-first with loud, explicit host fallback." v0.1 delivered the loud + explicit, but "host" still means "whatever UID you are, with no extra rules." A determined sqlmap RCE or wifi_deauth (which spawns `aircrack-ng`) on a host-fallback path is the worst-case blast radius.
- The supervisor (v0.2.A) and chokepoint (v0.1 phase 3) are both *observational* — they only stop calls that the model emits. v0.2.D is the *first* host-side kernel-enforced layer in the stack. The architectural promise is unchanged: the chokepoint is still the only policy gate; landlock/seccomp are below it in the trust hierarchy (a defense-in-depth layer that catches "the model mis-emitted + the user clicked confirm + the tool has a vuln" all at once).
- Threat model: a tool RCE in `aircrack-ng` running on the host should not give the attacker `cat ~/.config/gmft/config.toml` (the API key), `cat ~/.local/share/gmft/findings/*.json` (the engagement data), or the ability to `kill` arbitrary processes. Landlock + seccomp together provide a narrow enough allowlist to make that expensive.

**Host-environment note (verified 2026-06-17):**

- `/proc/sys/kernel/seccomp/actions_avail` shows `kill_process kill_thread trap errno user_notif trace log allow` — seccomp IS available on the host. But v0.2.D's decision 3 keeps seccomp as probe-only (v0.3 stretch).

**What ships in this plan:**

1. **`@gmft/tools`'s `runInSandbox`** gets three new optional fields on `RunOptions`: `fsAllowRead?: string[]`, `fsAllowWrite?: string[]`, `fsAllowMakeReg?: string[]`, `netMode?: 'none' | 'host' | 'inherit'`, `maxRuntimeMs?: number`. When the runner is in `host` mode AND the host supports landlock (kernel 5.13+, detected via `process.platform === 'linux'` + a feature probe), it applies landlock *and* a `seccomp(2)` filter via `child_process.spawn` `preExec` hook. When the runner is in `docker` mode, nothing changes (Docker already enforces).
2. **A `landlock` capability probe** (`packages/tools/src/shared/landlock.ts`) — single function `landlockAvailable(): { available: boolean, abiVersion: 1|2|3|4|5|6|7 | null, reason?: 'unsupported-platform' | 'kernel-too-old' | 'probe-failed' }`. Uses the `landlock` package's `getABI()` (which returns 1-7 on a kernel with landlock, throws otherwise). Exposed in `runner.ts` so the chokepoint can read it via a one-shot `runner.capabilities()` query at startup.
3. **A `seccomp` capability probe** (`packages/tools/src/shared/seccomp.ts`) — single function `seccompAvailable(): boolean`. Returns `false` on non-Linux. v0.2.D does NOT auto-apply seccomp; the probe exists so the chokepoint, the audit log, and the TUI can all read the same capability snapshot. (The BPF filter + `preExec` apply is v0.3 stretch — see ADR-0011.)
4. **A `runner.capabilities()` query** that returns `{ landlock: 'available' | 'unavailable' | 'denied', seccomp: 'available' | 'unavailable', docker: 'available' | 'unavailable' }` so the chokepoint, the TUI, and the audit log can all read the same snapshot.
5. **A new chokepoint rule `checkRequiresSandbox`** added at the bottom of the aggregator (after `checkTarget`, before the default `Allow`) — denies a tool call when:
   - the call has `destructive: true` or `requiresElevation: true` flag, AND
   - the configured runner mode would resolve to `host`, AND
   - `runner.capabilities()` reports `landlock: 'unavailable'` (i.e. no kernel support, no Docker), AND
   - the user has not set `GMFT_ALLOW_UNSANDBOXED_DESTRUCTIVE=true`
   The deny reason is a single line: `"host fallback for destructive/elevated tools requires Docker or kernel landlock (set GMFT_ALLOW_UNSANDBOXED_DESTRUCTIVE=true to override; not recommended)"`.
6. **An audit-log field** `runnerMode: 'docker' | 'host+landlock' | 'host' | 'unsandboxed'` recorded per tool invocation in the session log so a postmortem can see which calls ran in which mode. (Seccomp is omitted because v0.2.D does not auto-apply seccomp — see decision 3 and ADR-0011.)
7. **A `--sandbox` CLI flag** (`docker | host | auto`, default `auto`) wired to a `SandboxConfig.runner` field that overrides the existing `cfg.sandbox.mode` chain. Documents the three-way tradeoff (Docker = best, host+landlock = good, host = "do you have a *really* good reason?").
8. **TUI surface**: `StatusRail` grows a 4th field, "sandbox", with the resolved mode (docker / host+landlock / host) and a persistent ⚠ when mode is `host` (banner matches the existing "fell back" pattern from runner phase 3).
9. **CHANGELOG + ADR-0011** — `host-side-sandbox-hardening.md` — captures the decisions, the kernel probe strategy, the chokepoint rule order change, and the threat model. ADR-0003 is amended (footnote + superseding note) since this is the implementation of its deferral.

## Design decisions called out (push back if you disagree)

1. **The new chokepoint rule goes at the END of the aggregator, not at the top.** Rationale: a destructive tool that needs sandboxing is still a destructive tool; the existing destructive→confirm flow should fire first so the user is *asked*, and only when they say "yes" does the runner then refuse because there's no sandbox. Putting it at the top would deny a user-confirmed sqlmap run because the host is misconfigured, which is a worse UX than asking + denying. The rule order is now: `elevation → typeToConfirm → destructive → target → requiresSandbox → allow`. Documented in the chokepoint test as the new "rule order bottom" test case.

2. **Landlock implementation: use the `landlock` npm package (one new top-level dep).** The `landlock` package is the only realistic option for a clean `applyLandlock()` call from Node — the alternatives are a hand-rolled `sys_landlock_create_ruleset` / `sys_landlock_add_rule` / `sys_landlock_restrict_self` sequence via `dlopen`, which is brittle and adds the same `node-gyp` build cost. The `landlock` package is pre-1.0, has a single maintainer, and adds a native dep to a workspace that currently has zero native deps in `packages/tools`. Trade-off accepted: one new transitive dep for a clean API. ADR-0011 records the dep choice and the "if `landlock` goes unmaintained for 6+ months, fork it into `packages/landlock-shim/`" exit clause.

3. **Seccomp is probe-only in v0.2.D — no auto-apply.** v0.2.D ships the seccomp *probe* + a single end-to-end test that *if* seccomp is applied, it blocks a forbidden syscall. We do NOT auto-apply seccomp via `preExec` because (a) the BPF program needs hand-rolled BPF generation (~50 lines of BPF bytecode, easy to get wrong), and (b) the chokepoint's new rule does not require seccomp — it only requires landlock (or Docker). The apply path is documented as v0.3 stretch. The chokepoint rule does NOT require seccomp.

4. **The runner picks host+landlock mode when landlock is available, even if Docker is also available.** Rationale: Docker's container overhead (~200ms cold start per call) is a real cost for the user; if the kernel can do the same enforcement with no overhead, prefer it. The `--sandbox=docker` flag forces Docker when both are available. The `auto` default is: docker if available, else host+landlock if available, else host (with the chokepoint rule denying destructive calls). The StatusRail shows whatever the resolved mode is so the user knows.

5. **`runnerCapabilities()` is one-shot cached at module-load.** A `Map<name, boolean>` populated synchronously from the probes. Tests inject via `setCapabilitiesForTest()`. The chokepoint reads from this snapshot; the TUI reads from this snapshot. We do NOT probe on every call (the kernel call is cheap but the cache is cleaner and the test-injection pattern is simpler).

6. **The audit field is `runnerMode: 'docker' | 'host+landlock' | 'host' | 'unsandboxed'`.** Note `'host+seccomp'` is omitted because seccomp is not auto-applied in v0.2.D (decision 3). `'unsandboxed'` is the value the chokepoint records when it denied a call (so the audit log shows "denied because unsandboxed" not "denied for other reason"). v0.1 + v0.2.A logs that lack the field load with `runnerMode: undefined` — no migration needed.

7. **`--sandbox=host` is a deliberate opt-in, not the default.** The default is `auto`. The TUI's onboarding flow adds a 4th provider-card screen for "Sandbox mode" only when the user is on a fresh install (config missing) OR when they pass `--reconfigure`. The card shows the three modes with a one-line description each and the resolved capabilities (read from `runnerCapabilities()`).

8. **The chokepoint rule's deny reason must be a *single* line that fits in the StatusRail.** The reason is the *only* signal the user gets when their tool call is denied; multi-line reasons are clipped in the rail. The exact wording is the literal string in the chokepoint rule (Task 2.1, line 5 of the `denyReason` field).

9. **The TUI's persistent ⚠ is the SAME banner as the existing "host fallback" warning.** We do NOT add a new banner; the existing `<SandboxWarning>` component (added in phase 3, see `StatusRail.tsx`'s `if mode === 'host'` branch) is reused. The StatusRail's new "sandbox" field shows the mode; the banner stays the same.

  ```typescript
  /**
   * Landlock availability + application.
   *
   * Landlock is a Linux-only (5.13+) per-process unprivileged
   * access-control system. The probe reads the kernel's landlock
   * ABI version from `/proc/sys/kernel/landlock_abi_version`. The
   * application uses the `landlock` npm package to add rules and
   * enforce them on the calling process.
   *
   * v0.2.D-stretch (no auto-apply for seccomp): landlock is the
   * only kernel-enforced layer we apply automatically. Seccomp is
   * a separate probe + a single test.
   */

  import { createRequire } from 'node:module';
  const require_ = createRequire(import.meta.url);
  // The `landlock` package has no TS types; declare the surface we use.
  type LandlockLib = {
    constants: { [k: string]: bigint };
    getABI(): number;
    createRuleset(fsAccess: bigint | number, netAccess?: bigint | number, scoped?: bigint | number): number;
    addRule(fd: number, ruleType: bigint | number, ...args: unknown[]): void;
    setNoNewPrivs(): void;
    restrictSelf(fd: number, flags?: bigint | number): void;
    close(fd: number): void;
  };
  const landlock = require_('landlock') as LandlockLib;

  export interface LandlockStatus {
    available: boolean;
    abiVersion: 1 | 2 | 3 | 4 | 5 | 6 | 7 | null;
    reason?: 'unsupported-platform' | 'kernel-too-old' | 'probe-failed';
  }

  let cached: LandlockStatus | undefined;

  /** Probe once, cache the result. */
  export function landlockAvailable(): LandlockStatus {
    if (cached) return cached;
    if (process.platform !== 'linux') {
      cached = { available: false, abiVersion: null, reason: 'unsupported-platform' };
      return cached;
    }
    try {
      const abi = landlock.getABI();
      if (!Number.isInteger(abi) || abi < 1) {
        cached = { available: false, abiVersion: null, reason: 'kernel-too-old' };
      } else {
        // getABI returns 1..7; the package's source pins to the highest
        // supported ABI for the running kernel.
        cached = { available: true, abiVersion: abi as LandlockStatus['abiVersion'] };
      }
    } catch (err) {
      cached = {
        available: false,
        abiVersion: null,
        reason: 'probe-failed',
        ...(process.env.GMFT_DEBUG ? { _debug: String(err) } : {}),
      };
    }
    return cached;
  }

  /** Test seam: clear the cache so the next probe re-runs. */
  export function _resetLandlockForTest(): void {
    cached = undefined;
  }

  export interface ApplyLandlockOpts {
    /** Paths the child may read. */
    fsAllowRead?: string[];
    /** Paths the child may write. */
    fsAllowWrite?: string[];
    /** Paths the child may create new files/dirs in. */
    fsAllowMakeReg?: string[];
  }

  /**
   * Apply a landlock ruleset to the CURRENT process. Must be called
   * in the child's `preExec` hook (Node 21+) so the rules attach
   * to the child, not the parent.
   *
   * Throws `LANDLOCK_UNSUPPORTED` if the kernel doesn't support
   * landlock or the probe failed.
   */
  export function applyLandlock(opts: ApplyLandlockOpts): void {
    const status = landlockAvailable();
    if (!status.available) {
      const e = new Error(
        `landlock unavailable: ${status.reason ?? 'unknown'} (kernel 5.13+ with CONFIG_SECURITY_LANDLOCK required)`,
      );
      (e as Error & { code?: string }).code = 'LANDLOCK_UNSUPPORTED';
      throw e;
    }
    const readMask = opts.fsAllowRead && opts.fsAllowRead.length > 0
      ? landlock.constants.LANDLOCK_ACCESS_FS_READ_FILE
      : 0n;
    const writeMask = opts.fsAllowWrite && opts.fsAllowWrite.length > 0
      ? (landlock.constants.LANDLOCK_ACCESS_FS_WRITE_FILE |
         landlock.constants.LANDLOCK_ACCESS_FS_TRUNCATE_FILE)
      : 0n;
    const makeRegMask = opts.fsAllowMakeReg && opts.fsAllowMakeReg.length > 0
      ? landlock.constants.LANDLOCK_ACCESS_FS_MAKE_REG
      : 0n;
    const fsAccess = readMask | writeMask | makeRegMask;
    if (fsAccess === 0n) {
      // No rules means total lockout — refuse to apply (would block
      // the child from reading its own argv or libraries).
      throw new Error('applyLandlock called with no allowlist; refusing to lock down the child entirely');
    }
    const fd = landlock.createRuleset(fsAccess);
    try {
      const ruleType = landlock.constants.LANDLOCK_RULE_PATH_BENEATH;
      for (const p of opts.fsAllowRead ?? []) {
        landlock.addRule(fd, ruleType, readMask, p);
      }
      for (const p of opts.fsAllowWrite ?? []) {
        landlock.addRule(fd, ruleType, writeMask, p);
      }
      for (const p of opts.fsAllowMakeReg ?? []) {
        landlock.addRule(fd, ruleType, makeRegMask, p);
      }
      landlock.setNoNewPrivs();
      landlock.restrictSelf(fd);
    } finally {
      landlock.close(fd);
    }
  }
  ```

  *(The exact `landlock` npm package API needs to be verified during implementation; the import shape and `restrict()` call above are placeholders. The test for this file mocks the import.)*

  Create `packages/tools/src/shared/seccomp.ts`:

  ```typescript
  /**
   * Seccomp filter-mode availability probe.
   *
   * Seccomp is the syscall-filter companion to landlock. v0.2.D does
   * NOT auto-apply seccomp — it's a stretch that the BPF program
   * needs hand-rolled BPF generation. The probe exists so the
   * chokepoint can read it (and so a future v0.3 can wire it in).
   *
   * The probe reads `/proc/sys/kernel/seccomp/actions_avail`. A
   * non-empty value means the kernel supports seccomp filter mode;
   * an empty value or missing file means it doesn't.
   */

  import { existsSync, readFileSync } from 'node:fs';

  let cached: boolean | undefined;

  /**
   * Returns true iff the kernel supports seccomp filter mode AND
   * is willing to apply it to user processes. Probed via
   * `/proc/sys/kernel/seccomp/actions_avail` — a non-empty list
   * (e.g. "kill_process kill_thread trap errno user_notif trace
   * log allow") means filter mode is available; an empty file
   * means `CONFIG_SECCOMP=y` was not set or seccomp is disabled.
   *
   * Does NOT call `seccomp(2)` from the probe (that would
   * install a filter on the gmft process itself — wrong target).
   */
  export function seccompAvailable(): boolean {
    if (cached !== undefined) return cached;
    if (process.platform !== 'linux') {
      cached = false;
      return cached;
    }
    try {
      const path = '/proc/sys/kernel/seccomp/actions_avail';
      if (!existsSync(path)) {
        cached = false;
        return cached;
      }
      const content = readFileSync(path, 'utf-8').trim();
      // Empty content = seccomp not available; any non-empty content
      // (whitespace-separated action list) = filter mode supported.
      cached = content.length > 0;
    } catch {
      cached = false;
    }
    return cached;
  }

  export function _resetSeccompForTest(): void {
    cached = undefined;
  }
  ```

  ```typescript
  /**
   * The runner's one-shot capability snapshot. The chokepoint, the
   * TUI, and the audit log all read from this single source.
   *
   * Probes are cheap but the chokepoint path is sync; we cache the
   * result at module-load and expose `setCapabilitiesForTest` as
   * the test injection seam.
   */

  import { landlockAvailable, type LandlockStatus } from './landlock.js';
  import { seccompAvailable } from './seccomp.js';
  import { execSync } from 'node:child_process';

  export type LandlockCap = 'available' | 'unavailable' | 'denied';
  export type SeccompCap = 'available' | 'unavailable';
  export type DockerCap = 'available' | 'unavailable';

  export interface RunnerCapabilities {
    landlock: LandlockCap;
    landlockAbi: 1 | 2 | 3 | null;
    seccomp: SeccompCap;
    docker: DockerCap;
    /** What mode `auto` would resolve to right now. */
    resolvedAuto: 'docker' | 'host+landlock' | 'host' | 'unsandboxed';
  }

  let cached: RunnerCapabilities | undefined;

  function dockerProbe(): DockerCap {
    if (process.env.GMFT_SKIP_PREREQ === '1') return 'unavailable';
    try {
      // `docker info` is a more reliable presence check than `which docker`.
      execSync('docker info', { stdio: 'ignore', timeout: 2000 });
      return 'available';
    } catch {
      return 'unavailable';
    }
  }

  function compute(): RunnerCapabilities {
    const ll = landlockAvailable();
    const landlock: LandlockCap =
      ll.available ? 'available' : ll.reason === 'unsupported-platform' ? 'unavailable' : 'denied';
    const seccomp: SeccompCap = seccompAvailable() ? 'available' : 'unavailable';
    const docker: DockerCap = dockerProbe();
    let resolvedAuto: RunnerCapabilities['resolvedAuto'];
    if (docker === 'available') resolvedAuto = 'docker';
    else if (landlock === 'available') resolvedAuto = 'host+landlock';
    else resolvedAuto = 'host'; // chokepoint will deny destructive/elevated here
    return { landlock, landlockAbi: ll.abiVersion, seccomp, docker, resolvedAuto };
  }

  export function runnerCapabilities(): RunnerCapabilities {
    if (!cached) cached = compute();
    return cached;
  }

  /** Test seam: replace the cached snapshot. */
  export function setCapabilitiesForTest(caps: Partial<RunnerCapabilities>): void {
    cached = { ...runnerCapabilities(), ...caps };
  }

  /** Test seam: clear the cache. */
  export function _resetCapabilitiesForTest(): void {
    cached = undefined;
  }
  ```

  ```typescript
  /**
   * The "host fallback for destructive/elevated tools requires
   * Docker or kernel landlock" rule. v0.2.D's contribution to the
   * chokepoint.
   *
   * Order in the aggregator: LAST. The other 4 rules (elevation,
   * typeToConfirm, destructive, target) all fire first so the user
   * is *asked* (via the destructive confirm flow) before being
   * told the tool can't run.
   *
   * Fires when ALL of:
   *   - the call carries `destructive` or `requiresElevation`, AND
   *   - the resolved runner mode is `host` (no Docker, no
   *     landlock — i.e. `resolvedAuto === 'host'`), AND
   *   - the env does NOT have
   *     `GMFT_ALLOW_UNSANDBOXED_DESTRUCTIVE=true` set.
   *
   * The deny reason is a single line that fits in the StatusRail.
   */

  import type { Decision, ChokepointCall, ChokepointEnv } from './decision.js';
  import type { RunnerCapabilities } from '@gmft/tools';

  export function checkRequiresSandbox(
    call: ChokepointCall,
    env: ChokepointEnv & { runnerCapabilities: RunnerCapabilities; allowUnsandboxedDestructive: boolean },
  ): Decision | null {
    const isRisky = call.flags.includes('destructive') || call.flags.includes('requiresElevation');
    if (!isRisky) return null;
    if (env.runnerCapabilities.resolvedAuto !== 'host') return null;
    if (env.allowUnsandboxedDestructive) return null;
    return {
      kind: 'deny',
      reason:
        'host fallback for destructive/elevated tools requires Docker or kernel landlock ' +
        '(set GMFT_ALLOW_UNSANDBOXED_DESTRUCTIVE=true to override; not recommended)',
    };
  }
  ```

## Out of scope (deferred to v0.3+)

| Deferral | Notes |
|---|---|
| Seccomp auto-apply | v0.2.D is probe-only. v0.3 wires the BPF program into the runner's `preExec`. |
| macOS Seatbelt / sandbox-exec | macOS sandbox is per-app, not per-spawn; would need a different model. v0.2.D's chokepoint rule denies destructive tools on macOS by default. |
| Windows Job Objects | Same story; the runner's host path is a no-op on Windows and the chokepoint rule denies destructive tools. |
| AppArmor profiles | ADR-0003 mentions them as an alternative to landlock. v0.3 may add AppArmor *on top of* landlock for distros that ship an AppArmor policy. |
| Per-tool `fsAllowRead`/`fsAllowWrite` schemas | v0.2.D passes the allowlist via `RunOptions`; v0.3 may add a `Tool.fsAllowlist` field so the tool declares what it needs. |
| `--sandbox` onboarding card | Listed in decision 7 as a 4th provider-card screen; not in v0.2.D's scope. v0.3 stretch. |
| `runnerCapabilities()` cross-process caching | The current cache is per-process. v0.3 may move the probe to a one-shot script and cache the result in `~/.config/gmft/capabilities.json` so multiple invocations don't re-probe. |
